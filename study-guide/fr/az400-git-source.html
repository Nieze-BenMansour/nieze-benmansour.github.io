1. Introduction à DevOps

Introduction à DevOps — Microsoft Learn (~42 min · 15 unités · Débutant).

Objectifs d’apprentissage

  • Définir DevOps et ses principes (personnes, processus, produits pour une valeur continue).
  • Planifier la transformation : équipes, objectifs partagés, calendriers, structure agile.

  • Expliquer le contrôle de source et les fondamentaux Git en local.
  • Envisager une stratégie de gestion des licences.
  • Appliquer la réflexion sur le temps de cycle et l’apprentissage validé.

Qu’est-ce que DevOps ?

DevOps rapproche développement et opérations : équipes pluridisciplinaires, pratiques et outils partagés, responsabilité commune. On y trouve planification agile, intégration continue, livraison continue et supervision. DevOps est un parcours continu, pas un projet ponctuel.

Les résultats métier visés incluent une fréquence de déploiement plus élevée, un délai d’exécution plus court, un MTTR plus faible et moins d’échecs de changement — donc un time-to-market plus rapide et moins d’incidents.

Temps de cycle, OODA et apprentissage validé

La boucle OODA (Observer, Orienter, Décider, Agir) s’applique bien à la livraison : observer la télémétrie, orienter avec des expériences, décider des priorités, agir en livrant et en mesurant. Le temps de cycle va de l’idée ou du commit au retour production. Soyez guidés par les données sans paralysie : fail fast, apprentissage validé à chaque itération.

Azure DevOps, GitHub, Git

Le module présente Azure DevOps (Boards, Repos, Pipelines, Test Plans, Artifacts) et GitHub. Le contrôle de source trace les versions ; Git est distribué ; Azure Repos et GitHub hébergent les remotes et les flux de demandes de tirage.

2. Planifier l’agilité avec GitHub Projects et Azure Boards

Planifier l’agilité avec GitHub Projects et Azure Boards — Microsoft Learn (~1 h 34 · 14 unités · Avancé).

Objectifs d’apprentissage

  • Décrire GitHub Projects et Azure Boards.
  • Lier Azure Boards et GitHub.
  • Configurer et gérer GitHub Projects et les tableaux.
  • Personnaliser les vues de projet.

GitHub Projects

GitHub Projects organisent et visualisent le travail (tableaux, roadmaps). Les tableaux suivent issues et pull requests avec colonnes et automatisation.

Azure Boards

Azure Boards assure le suivi du travail Agile, Scrum, Kanban ; requêtes, tableaux de bord, Delivery Plans multi-équipes.

Lier GitHub à Azure Boards

L’intégration connecte commits, branches et PR aux éléments de travail pour la traçabilité (voir GitHub et Azure Boards). Utilisez les conventions AB#<id> dans les commits.

3. Concevoir et implémenter des stratégies et flux de branches

Concevoir et implémenter des stratégies et flux de branches — Microsoft Learn (~1 h 32 · 10 unités · Avancé).

Objectifs d’apprentissage

  • Décrire les flux de branchement Git.
  • Implémenter des branches de fonctionnalité.
  • Implémenter GitHub Flow.
  • Créer un fork d’un dépôt.

Choisir un flux

Adaptez le flux à la taille d’équipe, au rythme de release et aux barrières qualité. GitHub Flow pour la livraison continue ; branches de fonctionnalité pour isoler avant fusion ; branches de release pour stabiliser ; fork pour l’open source ou des frontières d’accès strictes.

ModèleAdapté quand
GitHub FlowUne branche principale toujours déployable ; branches courtes et PR ; automatisation forte.
Branche de fonctionnalitéFonctionnalités parallèles, branche principale stable, revue avant fusion.
Branche de releaseReleases planifiées, QA prolongée.
ForkContributeurs externes ; fusion depuis les forks.

4. Collaborer avec des demandes de tirage dans Azure Repos

Collaborer avec des demandes de tirage dans Azure Repos — Microsoft Learn (~30 min · 5 unités · Avancé).

Prérequis (Learn) : bases du branchement Git ; Azure DevOps ; concepts de revue de code.

Objectifs d’apprentissage

  • Mettre en œuvre des flux de demandes de tirage pour la collaboration à l’échelle.
  • Concevoir des revues qui améliorent qualité et transfert de connaissances.

  • Exploiter l’intégration Azure Repos pour les modèles inter-équipes.

Rôle des demandes de tirage

Les PR sont le lieu d’intégration sécurisée : revue, discussion, automatisation (build/test), règles de fusion.

Azure Repos en entreprise

Azure Repos s’intègre à Microsoft Entra ID, se connecte à Azure Boards, Azure Pipelines, Azure Test Plans. Les stratégies de branche encodent revues minimales, builds obligatoires, résolution des commentaires.

5. Explorer les hooks Git

Explorer les hooks Git — Microsoft Learn (~14 min · 5 unités · Avancé).

Prérequis (Learn) : pratique Git solide ; scripts ; bases CI ; sensibilisation sécurité.

Objectifs d’apprentissage

  • Créer des hooks qui appliquent la qualité pour l’équipe.
  • Exécuter des contrôles avant et après commit.

7. Gérer et configurer les référentiels

Gérer et configurer les référentiels — Microsoft Learn (~48 min · 15 unités · Avancé).

Objectifs d’apprentissage

  • Travailler avec de gros dépôts Git.
  • Utiliser VFS for Git si pertinent.
  • Utiliser Git LFS pour les gros binaires.
  • Nettoyer les données du dépôt.
  • Gérer et automatiser les notes de publication GitHub.

Pourquoi les dépôts grossissent

Historique long et gros binaires : clone superficiel, clone partiel, sparse-checkout réduisent la charge locale.

Git LFS

Git LFS stocke les gros fichiers hors du magasin d’objets classique. Voir la documentation GitHub.

8. Identifier la dette technique

Identifier la dette technique — Microsoft Learn (10 unités · Avancé).

Objectifs d’apprentissage

  • Repérer et gérer la dette technique.
  • Utiliser des outils de qualité pour améliorer le logiciel.
  • Planifier des revues de code efficaces.
  • Comprendre complexité et métriques qualité.

Qu’est-ce que la dette technique ?

La dette technique est le coût futur des raccourcis d’aujourd’hui — comme un intérêt financier. Elle réduit la vélocité et augmente les défauts ; il faut la rembourser régulièrement.

Mesure et outils

Analyse automatisée (barrières qualité en CI), SonarCloud / SonarQube, GitHub Advanced Security, revue humaine pour prioriser.
