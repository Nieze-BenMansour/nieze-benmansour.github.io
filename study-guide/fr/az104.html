1. Gérer les identités et la gouvernance Azure

Poids de l'examen : 20 à 25 %

Décrire les principaux composants architecturaux d'Azure

Ce que propose Azure

Azure est un ensemble de services cloud en constante expansion pour répondre aux défis des entreprises. Vous pouvez créer, gérer et déployer des applications sur un réseau mondial à l'aide de vos outils et frameworks préférés. Offres Azure innovation illimitée (applications intelligentes avec IA et services avancés), gestion de plateforme unifiée (infrastructure, données, analyses et IA en un seul endroit), et confiance (sécurité et conformité). Azure fournit plus de 100 services, depuis l'exécution d'applications existantes sur des machines virtuelles jusqu'à l'IA/ML (vision, audition, parole), en passant par le stockage évolutif et les solutions nécessitant une évolutivité dans le cloud. De nombreuses équipes commencent par déplacer les applications vers des machines virtuelles, mais le cloud permet bien plus (par exemple, stockage dynamique, robots, réalité mixte).

Premiers pas avec les comptes Azure

Pour utiliser les services Azure, vous avez besoin d'un Abonnement Azure . Créez un compte Azure (par exemple sur le site Web Azure ou via un représentant Microsoft ou un partenaire fournisseur de solutions cloud) ; un abonnement est créé pour vous. Vous pouvez créer des abonnements supplémentaires (par exemple un par département : développement, marketing, ventes). Les ressources sont créées au sein des abonnements.

Compte Azure gratuit : Accès gratuit aux produits Azure populaires pendant 12 mois ; un crédit pour les 30 premiers jours ; Plus de 25 produits toujours gratuits. Inscrivez-vous avec un téléphone, une carte de crédit (vérification uniquement, sans frais jusqu'à la mise à niveau) et un compte Microsoft ou GitHub. Bon pour explorer.

Compte étudiant gratuit Azure : Accès gratuit à certains services pendant 12 mois ; crédit pour 12 mois ; outils de développement gratuits. Pour les étudiants ; Crédit de 100 $ ; aucune carte de crédit requise. De nombreux exercices d'apprentissage sont apportez votre propre abonnement (BYOS) ; effectuez les étapes de nettoyage pour éviter des coûts inattendus.

Infrastructure physique Azure

L’infrastructure physique d’Azure commence par centres de données (installations avec racks, alimentation, refroidissement, mise en réseau). Les centres de données ne sont pas directement accessibles ; ils sont regroupés en Régions et Zones de disponibilité pour la résilience et la fiabilité.

Régions — Une zone géographique contenant un ou plusieurs datacenters mis en réseau avec une faible latence. Lorsque vous déployez une ressource, vous choisissez généralement une région. Certains services ou fonctionnalités de VM ne sont disponibles que dans certaines régions ; certains services sont mondial (pas de sélection de région), par ex. ID Microsoft Entra, Azure Traffic Manager, Azure DNS.

Zones de disponibilité — Centres de données physiquement séparés au sein d'une région. Chaque zone dispose d'une alimentation, d'un refroidissement et d'un réseau indépendants ; si une zone échoue, les autres continuent. Connecté par fibre haut débit. Minimum de trois zones dans toutes les régions compatibles avec les zones de disponibilité ; toutes les régions ne les soutiennent pas. À utiliser pour les machines virtuelles, les disques gérés, les équilibreurs de charge et la base de données SQL. Catégories : zonal (épinglé à une zone), redondant de zone (répliqué à travers les zones par la plateforme), non régional (disponible dans toutes les zones géographiques, résilient aux pannes de zone et de région).

Paires de régions — La plupart des régions sont associées à d'autres régions situées dans la même zone géographique (par exemple, les États-Unis, l'Europe, l'Asie), distantes d'au moins 300 milles. Permet la réplication et réduit l'impact des sinistres, des pannes ou des pannes de réseau ; une région peut basculer vers sa paire. Tous les services ne se répliquent pas automatiquement ou ne basculent pas : le client doit effectuer la configuration si nécessaire. Avantages : lors d'une panne majeure d'Azure, une région par paire est prioritaire pour la restauration ; les mises à jour planifiées sont déployées une par une dans les régions appariées ; les données restent dans la même zone géographique (sauf le sud du Brésil). La plupart des paires sont bidirectionnelles (par exemple, l'Ouest des États-Unis et l'Est des États-Unis se soutiennent mutuellement) ; certains sont à sens unique (par exemple, le secondaire de l’Inde de l’Ouest est l’Inde du Sud, mais le secondaire de l’Inde du Sud est l’Inde centrale ; le secondaire du Sud du Brésil est le Centre-Sud des États-Unis).

Régions souveraines — Instances isolées d'Azure pour des raisons de conformité ou juridiques. Exemples : DoD Central des États-Unis, gouvernement américain de Virginie, gouvernement américain de l'Iowa (isolement physique et logique pour le gouvernement américain ; personnel contrôlé ; conformité accrue) ; Chine Est, Chine Nord (via le partenariat Microsoft-21Vianet ; Microsoft n'exploite pas directement les centres de données).

Infrastructure de gestion Azure

L'infrastructure de gestion comprend ressources , groupes de ressources , abonnements , et groupes de gestion .

Ressources et groupes de ressources - UN ressource est la brique de base (VM, réseaux virtuels, bases de données, services cognitifs, etc.). Vous devez placer chaque ressource dans un groupe de ressources . Une ressource ne peut appartenir qu’à un seul groupe de ressources à la fois ; certaines ressources peuvent être déplacées. Les groupes de ressources ne peuvent pas être imbriqués. Les actions sur un groupe de ressources s'appliquent à toutes les ressources qu'il contient (suppression, contrôle d'accès). Utilisez des groupes de ressources pour organiser par cycle de vie (par exemple, supprimer l'intégralité de l'environnement de développement) ou par accès (par exemple, attribuer un accès au niveau du groupe de ressources).

Abonnements — Une unité de gestion , facturation , et échelle . Ils organisent des groupes de ressources et facilitent la facturation. Un abonnement donne un accès authentifié et autorisé à Azure et permet le provisionnement. Un abonnement est lié à un compte Azure (identité dans Microsoft Entra ID ou un annuaire de confiance). Un compte peut avoir plusieurs abonnements. Deux types de limites : limite de facturation (comment le compte est facturé ; factures séparées par abonnement) ; limite de contrôle d'accès (politiques d'accès au niveau de l'abonnement ; par exemple, différents départements, différentes politiques). Créez des abonnements supplémentaires à des environnements distincts (développement/test, conformité), une structure organisationnelle ou un suivi de facturation/coûts.

Groupes de gestion — Un périmètre au-dessus des abonnements. Organisez les abonnements en groupes de gestion et appliquer la gouvernance (politiques, RBAC). Les abonnements héritent des conditions de leur groupe d'administration (comme les groupes de ressources héritent des abonnements). Permet une gestion à l’échelle de l’entreprise. Cas d'utilisation : appliquez une stratégie (par exemple, limiter les régions de VM) à un groupe de gestion afin que tous les abonnements descendants en héritent ; attribuez une fois RBAC au groupe d’administration pour accéder à plusieurs abonnements. Limites : jusqu'à 10 000 groupes de gestion par répertoire ; profondeur des arbres jusqu'à six niveaux (hors root et abonnement) ; chaque groupe d’administration et abonnement n’a qu’un seul parent.

Utilisateurs et groupes

Présentation de l’ID Microsoft Entra

Identifiant Microsoft Entra est un logiciel géré par Microsoft Service d'annuaire PaaS dans le cloud (pas IaaS ; vous ne l’exécutez ni ne le maintenez). Il assure la gestion des identités et des accès pour les environnements cloud et hybrides. Contrairement à AD DS (sur site, contrôleurs de domaine, GPO), Entra ID est conçu pour les applications Web et les services cloud, avec des fonctionnalités telles que l'authentification multifacteur, la protection de l'identité et la réinitialisation du mot de passe en libre-service.

Utilisez Entra ID pour : configurer l'accès aux applications ; SSO vers les applications SaaS ; gérer les utilisateurs, les groupes et le provisionnement ; fédération; gestion des identités ; détecter une activité de connexion irrégulière ; MAE ; étendre l'AD sur site au cloud ; Mandataire d'application ; Accès conditionnel.

Licence : UN Gratuit Le niveau est inclus avec chaque nouvel abonnement Azure (et avec Microsoft 365/Intune). Basique et Prime ajouter des fonctionnalités avancées. La création d'un abonnement Azure avec un compte Microsoft vous donne un locataire Entra par défaut nommé Répertoire par défaut .

Locataires Microsoft Entra

L'ID d'entrée est multi-locataire par conception ; chaque instance de répertoire est isolée. UN locataire est l’instance Entra d’une organisation (ou techniquement, un répertoire Entra). Un L'abonnement Azure est lié à exactement un Entrer le locataire à la fois ; ce lien vous permet d'accorder RBAC sur l'abonnement aux utilisateurs, groupes et applications de ce locataire. Le même locataire peut être associé à plusieurs abonnements Azure .

Chaque locataire obtient un domaine DNS par défaut : <unique-prefix>.onmicrosoft.com . Vous pouvez ajouter domaines personnalisés . Le locataire est le frontière de sécurité et conteneur pour les utilisateurs, les groupes et les applications.

Schéma Microsoft Entra

Moins de types d’objets qu’AD DS : non ordinateur classe (il y a un appareil classe; le processus de jointure diffère de AD DS). Non UO classe : pas de hiérarchie personnalisée de conteneurs ; utiliser appartenance à un groupe plutôt. Le schéma est extensible et réversible.

Pas de GPO traditionnels ; Utilisations d'Entra gestion moderne . Points forts : services d'annuaire, stockage/publication des données utilisateur/appareil/application, authentification et autorisation.

Applications : Application = définition de l'application ; servicePrincipal = instance de cette application dans un locataire. Une application peut être utilisée dans plusieurs locataires en créant un principal de service par locataire (créé lorsque vous enregistrez l'application dans ce locataire).

EntraID et AD DS

AD DS (sur site) : Annuaire hiérarchique basé sur X.500 ; DNS pour localiser les contrôleurs de domaine ; LDAP pour les requêtes/gestions ; Kerberos pour l'authentification ; Unités d'organisation et objets de stratégie de groupe pour la gestion ; objets informatiques (machines jointes au domaine) ; approbations entre domaines. Fait partie de la suite Windows Active Directory (AD CS, AD LDS, AD FS, AD RMS). Vous pouvez exécuter AD DS sur des machines virtuelles Azure pour des raisons d'évolutivité et de disponibilité. pas utilisez l'identifiant Entra. Lors du déploiement d'AD DS sur une machine virtuelle Azure, utilisez des disques de données distincts (et non C :) pour la base de données AD, les journaux et sysvol ; ensemble Préférence de cache d'hôte = Aucune pour ces disques.

Identifiant Microsoft Entra (nuage) : Solution d'identité pour applications basées sur Internet (HTTP/HTTPS). Multi-locataire annuaire; plat structure utilisateur/groupe : pas d'unité d'organisation ni de GPO. Pas de LDAP : utilise API REST via HTTP/HTTPS. Pas de Kerberos : utilise SAML, WS-Federation, OpenID Connect pour l'authentification et OAuth pour autorisation. Comprend la fédération ; de nombreux services tiers (par exemple Facebook) se fédèrent avec Entra ID.

Entra ID comme répertoire pour les applications cloud

Les services cloud (Microsoft 365, Intune, etc.) nécessitent un annuaire cloud pour l'authentification. En utilisant un locataire Entra pour tous les services cloud Microsoft (Microsoft 365, Azure, Dynamics 365, Intune) donne une couche d'identité unique. Entra ID fournit une authentification centralisée pour les applications Azure, y compris avec d'autres fournisseurs d'identité ou AD DS sur site, et SSO pour Microsoft et les applications tierces (par exemple Facebook, Google, Yahoo).

Pour les applications personnalisées, vous pouvez activer Authentification Microsoft Entra pour Applications Web Azure App Service de la Authentification/Autorisation dans le portail Azure : sélectionnez le locataire Entra afin que seuls les utilisateurs de ce répertoire puissent accéder à l'application. Vous pouvez définir différentes options d'authentification par emplacement de déploiement .

Créer, configurer et gérer des identités

Créer, configurer et gérer des utilisateurs

Chaque utilisateur ayant besoin d'accéder aux ressources Azure a besoin d'un Compte utilisateur Azure . Un compte utilisateur contient les informations nécessaires pour authentifier l'utilisateur lors de la connexion. Après l'authentification, Microsoft Entra ID émet un jeton d'accès utilisé pour autoriser l'utilisateur et déterminer à quelles ressources il peut accéder et quelles actions il peut effectuer.

Vous gérez les objets utilisateur depuis le Identifiant Microsoft Entra tableau de bord dans le portail Azure. Vous travaillez avec un répertoire à la fois ; utiliser le Annuaire + Abonnement panneau ou le Changer de répertoire dans la barre d'outils pour changer de répertoire.

Afficher les utilisateurs

Pour afficher les utilisateurs : accédez à Identité → Utilisateurs → Tous les utilisateurs . Le Type d'utilisateur La colonne indique si chaque utilisateur est un Membre ou un Invité .

Types d'identité d'utilisateur

Microsoft Entra ID définit les utilisateurs de trois manières principales :

  • Identités cloud — Existe uniquement dans Microsoft Entra ID. Exemples : comptes d'administrateur et utilisateurs que vous gérez directement. Source: Identifiant Microsoft Entra ou Répertoire Microsoft Entra externe (l'utilisateur vit dans un autre locataire Entra mais a besoin d'accéder aux ressources de ce répertoire). Une fois supprimés du répertoire principal, ces comptes sont supprimés.
  • Identités synchronisées par annuaire — Exister dans Active Directory sur site. Microsoft EntraConnect les synchronise dans Azure. Source: Windows Server AD .
  • Utilisateurs invités — Exister en dehors d’Azure. Exemples : comptes d'autres fournisseurs de cloud et comptes Microsoft (par exemple Xbox LIVE). Source: Utilisateur invité . Utilisation pour des fournisseurs ou entrepreneurs externes ; supprimer le compte lorsque l’accès n’est plus nécessaire.

Gérer les utilisateurs externes

Lors de l'examen et de Microsoft Learn, utilisateurs externes sont généralement utilisateurs invités (Collaboration B2B) : utilisateurs d'une autre organisation ou fournisseur d'identité qui ont besoin d'accéder à vos applications et ressources. Gérez-les via Identité → Utilisateurs → Tous les utilisateurs (filtrer par type d'utilisateur = Invité) ou Identités externes → Tous les utilisateurs externes dans le centre d'administration Microsoft Entra. Invitez des invités par e-mail ; ils utilisent l'invitation et se connectent avec leurs propres informations d'identification. Supprimez ou révoquez l'accès des invités lorsqu'il n'est plus nécessaire pour que l'accès reste précis et vérifiable.

Autoriser les utilisateurs à réinitialiser leur mot de passe avec la réinitialisation de mot de passe en libre-service Microsoft Entra (SSPR)

Réinitialisation du mot de passe en libre-service (SSPR) permet aux utilisateurs de réinitialiser leur mot de passe dans un navigateur ou à partir de l'écran de connexion Windows lorsqu'ils ne sont pas connectés, qu'ils ont oublié leur mot de passe ou qu'il a expiré. Ils retrouvent l'accès à Azure, Microsoft 365 et à d'autres applications qui utilisent Microsoft Entra ID. SSPR réduit la charge du service d’assistance et la perte de productivité. Tout utilisateur connecté peut changement leur mot de passe dans n'importe quelle édition Entra ; réinitialiser lorsqu'il n'est pas connecté, il nécessite SSPR (voir licence ci-dessous).

Comment fonctionne le SSPR : L'utilisateur accède au portail de réinitialisation du mot de passe ou sélectionne « Impossible d'accéder à votre compte » lors de la connexion. Étapes : (1) Localisation — langue de la page selon les paramètres régionaux du navigateur ; (2) Vérification — l'utilisateur saisit son nom d'utilisateur et transmet le CAPTCHA ; (3) Authentification — l'utilisateur prouve son identité avec la ou les méthodes requises ; (4) Réinitialisation du mot de passe — saisir et confirmer le nouveau mot de passe ; (5) Notification — confirmation à l'utilisateur. Vous pouvez personnaliser l'expérience (par exemple le logo de l'entreprise).

Créer, configurer et gérer des groupes

UN Groupe Microsoft Entra organise les utilisateurs et simplifie la gestion des autorisations. Les propriétaires de ressources (ou propriétaires de répertoires) attribuent des autorisations d'accès à l'ensemble du groupe plutôt qu'à chaque utilisateur. Les groupes définissent une limite de sécurité ; vous ajoutez ou supprimez des utilisateurs pour accorder ou refuser l'accès avec un minimum d'effort. Entra ID prend également en charge adhésion basée sur des règles (par exemple par département ou titre de poste).

Types de groupes

Microsoft Entra ID prend en charge deux types de groupes :

  • Groupes de sécurité — Le plus courant. Utilisé pour gérer l’accès des membres et des ordinateurs aux ressources partagées. Créez un groupe de sécurité pour une stratégie spécifique et attribuez des autorisations à tous les membres en même temps plutôt qu'à chaque utilisateur. Nécessite un administrateur Microsoft Entra.
  • Groupes Microsoft 365 — Activer la collaboration : boîte aux lettres partagée, calendrier, fichiers, site SharePoint, etc. Peut inclure des personnes extérieures à l'organisation. Disponible pour les utilisateurs et les administrateurs.

Afficher les groupes disponibles

Afficher tous les groupes via Identité → Groupes dans le centre d'administration Microsoft Entra. Un nouveau déploiement Entra ID ne comporte aucun groupe par défaut.

Type d'adhésion

Type d'adhésion définit comment les membres sont ajoutés :

  • Attribué — Les membres sont ajoutés et gérés manuellement.
  • Dynamique — Les membres sont ajoutés par des règles ( groupe dynamique ). Le groupe est toujours un groupe de sécurité ou un groupe Microsoft 365 ; seule l'adhésion est basée sur des règles.

Groupes dynamiques

Dans un groupe dynamique , l'adhésion est recalculée par une formule à chaque utilisation du groupe. Tout objet d'annuaire dont les attributs correspondent au filtre du groupe est inclus. Si les propriétés d’un utilisateur sont modifiées pour correspondre au filtre, il peut devenir membre par inadvertance. Un provisionnement de compte bien défini et cohérent réduit ce risque. Les groupes dynamiques nécessitent Entra ID P1 ou P2.

Configurer et gérer l'enregistrement des appareils

L'informatique doit trouver un équilibre entre la productivité des utilisateurs (n'importe quel appareil, n'importe où) et la protection des actifs de l'organisation. Microsoft Entra ID fournit identité de l'appareil afin que vous puissiez gérer et sécuriser les accès ; des outils comme Microsoft Intune s'appuyer sur cette identité pour la conformité et la configuration. Entra ID active l'authentification unique sur les appareils, les applications et les services.

Appareils enregistrés Microsoft Entra - Pour BYOD et mobile . L'appareil est enregistré sur Entra ID ; l'utilisateur se connecte avec un compte local/personnel et joint un compte organisationnel pour les ressources de travail. Aucune connexion organisationnelle à l’appareil lui-même. Utiliser Accès conditionnel et MDM (par exemple Intune) pour exiger des appareils conformes. L'inscription peut avoir lieu lors de l'accès à une application professionnelle pour la première fois ou via les paramètres Windows. Scénario typique : l'utilisateur accède à la messagerie électronique et aux outils RH depuis un ordinateur personnel ; après avoir enregistré l’appareil et satisfait à la conformité Intune, l’accès conditionnel accorde l’accès.

Utilisez-le lorsque : vous avez besoin d'un accès conditionnel sur site (par exemple, une application derrière ADFS qui vérifie l'identité de l'appareil via des règles de réclamation), ou lorsque vous utilisez Windows Hello Entreprise dans des scénarios hybrides ou fédérés (WHFB nécessite une réécriture de l’appareil).

Gérer les licences

Les services cloud payants Microsoft (Microsoft 365, Enterprise Mobility + Security, Dynamics 365, etc.) nécessitent licences au niveau utilisateur . Microsoft Entra ID stocke l’état d’attribution de licence. Les licences sont gérées via des portails de gestion (Office ou Azure) ou PowerShell. Auparavant, les licences ne pouvaient être attribuées que par utilisateur, ce qui rendait les modifications à grande échelle (par exemple, déménagements de service, arrivées/départs) fastidieuses et nécessitait souvent des scripts PowerShell personnalisés.

Licences basées sur un groupe — Attribuer une ou plusieurs licences de produit à un groupe ; Entra ID attribue des licences à tous les membres, ajoute des licences lorsque les utilisateurs rejoignent le groupe et les supprime lorsqu'ils le quittent. Cela réduit ou élimine l’automatisation des licences par utilisateur via PowerShell.

Exigences de licence pour les licences basées sur des groupes :

  • Payant ou essai Microsoft EntraID Premium P1 ou plus, ou Office 365 Entreprise E3 ou plus.
  • Vous devez avoir suffisamment de licences pour chaque membre unique de tout groupe auquel des licences ont été attribuées. Vous n'êtes pas obligé d'attribuer chaque membre individuellement, mais le locataire doit disposer d'au moins une licence par membre unique (par exemple, 1 000 membres dans des groupes sous licence → 1 000 licences requises).

Caractéristiques:

  • Les licences peuvent être attribuées à n'importe quel groupe de sécurité (synchronisé sur site via Entra Connect, cloud uniquement ou groupes dynamiques).
  • Lors de l'attribution d'un produit à un groupe, vous pouvez désactiver des plans de service spécifiques (par exemple, attribuez Microsoft 365 mais désactivez Yammer jusqu'à ce que l'organisation soit prête).
  • Pris en charge : tous les produits Microsoft 365, Enterprise Mobility + Security, Dynamics 365 et autres services cloud Microsoft sous licence au niveau utilisateur.
  • Les licences basées sur le groupe sont disponibles via le Centre d'administration Microsoft 365 (pas le portail Azure). Les modifications de licence résultant des mises à jour d’adhésion prennent généralement effet en quelques minutes.
  • Un utilisateur peut être dans plusieurs groupes avec des licences et peut aussi avoir directement affecté licences. L'état final est la combinaison de tout ; L'attribution de produits en double consomme la licence une seule fois.
  • Si Entra ID ne peut pas attribuer de licence (par exemple, pas assez de licences, services en conflit), les administrateurs peuvent voir quels utilisateurs n'ont pas été entièrement traités et prendre des mesures correctives.
  • Lieu d'utilisation doit être défini dans le profil utilisateur avant l’attribution – certains services ne sont pas disponibles dans tous les emplacements. Pour l'affectation de groupe, les utilisateurs sans emplacement d'utilisation héritent de l'emplacement du répertoire ; définissez l'emplacement d'utilisation lors de la création de l'utilisateur lorsque vous avez des utilisateurs à plusieurs emplacements.

Explorez la création automatique d'utilisateurs

Le provisionnement automatique des utilisateurs crée et met à jour les comptes d'utilisateurs (et de groupes) dans Microsoft Entra ID ou les applications cibles basées sur un système source, par exemple un HCM (Gestion du Capital Humain) système. Lorsque des employés sont ajoutés ou mis à jour dans RH, les comptes et les attributs sont synchronisés ; lorsqu'ils quittent ou changent de rôle, les comptes peuvent être mis à jour ou supprimés. Garder les systèmes d’identité synchronisés avec les RH réduit les comptes obsolètes et les risques d’accès.

SCIM (Système de gestion des identités inter-domaines) est une norme ouverte permettant d'automatiser l'échange d'informations sur l'identité des utilisateurs entre les domaines d'identité et les systèmes informatiques. Utilisations de Microsoft Entra ID SCIM2.0 pour le provisionnement automatique : le Service de provisionnement Microsoft Entra se connecte à l'application Point de terminaison SCIM , utilise le schéma d'objet utilisateur SCIM et les API REST, et automatise le provisionnement et le déprovisionnement des utilisateurs et des groupes. Si les utilisateurs sont automatiquement déprovisionnés d'Entra ID lorsqu'ils sont supprimés du système RH, vous réduisez la fenêtre d'accès non autorisé à partir de comptes orphelins.

Composants :

  • Système HCM — Applications et technologies qui soutiennent les processus RH tout au long du cycle de vie des employés (embauche, changement de rôle, congés). Source de vérité pour qui devrait y avoir accès.
  • Service de provisionnement Microsoft Entra — Utilise SCIM 2.0 ; se connecte au point de terminaison SCIM de l'application et utilise le schéma d'utilisateur/groupe SCIM et les API REST pour créer, mettre à jour et supprimer des utilisateurs et des groupes dans le système cible.
  • Identifiant Microsoft Entra — Référentiel d'utilisateurs ; gère le cycle de vie des identités et de leurs droits. Il peut s'agir de la source (provisionnement vers des applications) ou de la cible (provisionné à partir de HCM).
  • Système cible — Application ou système qui expose un point de terminaison SCIM et fonctionne avec le provisionnement Microsoft Entra pour permettre le provisionnement et le déprovisionnement automatiques des utilisateurs et des groupes.

Qu’est-ce qu’Azure RBAC ?

Identifiant Microsoft Entra et Azure RBAC travaillez ensemble afin que lorsque les gens quittent l'organisation, ils perdent l'accès aux ressources cloud, et afin que vous puissiez équilibrer l'autonomie des équipes avec une gouvernance centrale (par exemple, les équipes gèrent les machines virtuelles tandis que vous contrôlez les réseaux). Chaque abonnement Azure est associé à un annuaire Microsoft Entra ; les utilisateurs, les groupes et les applications de ce répertoire sont utilisés pour l'authentification unique et la gestion des accès. Avec Microsoft EntraConnect , les comptes AD sur site peuvent gérer les abonnements Azure ; la désactivation d'un compte sur site supprime l'accès aux abonnements Azure connectés.

Azure RBAC est un autorisation système construit sur Gestionnaire de ressources Azure qui fournit une gestion fine des accès. Vous accordez l'accès dont les utilisateurs ont besoin, par ex. un utilisateur gère les machines virtuelles dans un abonnement, un autre gère les bases de données SQL. Vous accordez l'accès en attribuant un Rôle Azure aux utilisateurs, groupes ou applications à un portée : groupe d'administration, abonnement, groupe de ressources ou ressource unique. Un rôle au niveau d'une étendue parent accorde l'accès aux étendues enfants (par exemple, l'accès à un groupe de ressources accorde l'accès à toutes les ressources qu'il contient). Une ressource appartient à un seul groupe de ressources ; un groupe de ressources à un seul abonnement ; un abonnement à un locataire Entra.

Que pouvez-vous faire avec Azure RBAC ?

  • Autorisez un utilisateur à gérer les machines virtuelles et un autre à gérer les réseaux virtuels dans le cadre d'un abonnement.
  • Autoriser un groupe DBA à gérer des bases de données SQL dans un abonnement.
  • Autoriser un utilisateur à gérer toutes les ressources d'un groupe de ressources (VM, sites Web, sous-réseaux).
  • Autoriser une application à accéder à toutes les ressources d'un groupe de ressources.

Dans le portail Azure, utilisez le Contrôle d'accès (IAM) sur les groupes d'administration, les abonnements, les groupes de ressources ou les ressources pour voir qui a accès et pour accorder ou supprimer l'accès.

Comment fonctionne Azure RBAC ?

L'accès est contrôlé par attributions de rôles . Une attribution de rôle comporte trois éléments : directeur de la sécurité (OMS), définition du rôle (quoi), et portée (où).

1. Responsable de la sécurité (qui) — L'utilisateur, le groupe ou l'application (principal du service) auquel vous accordez l'accès.

2. Définition du rôle (quoi) — Un ensemble d'autorisations (souvent appelé rôle). Répertorie les opérations que le rôle peut effectuer (lecture, écriture, suppression). Peut être large (par exemple, propriétaire) ou spécifique (par exemple, contributeur de machine virtuelle).

Rôles intégrés (exemples) :

Rôle Description
Propriétaire Accès complet à toutes les ressources, y compris le droit de déléguer l'accès à d'autres.
Donateur Créer et gérer tous les types de ressources Azure ; ne peut pas accorder l’accès à d’autres.
Lecteur Affichez les ressources Azure existantes.
Administrateur d'accès utilisateur Gérez l’accès des utilisateurs aux ressources Azure.

Si les rôles intégrés sont insuffisants, créez rôles personnalisés .

3. Portée (où) — Le niveau auquel l'accès s'applique : groupe d'administration, abonnement, groupe de ressources ou ressource. Les étendues ont une relation parent-enfant ; l'accès au niveau d'une portée parent est hérité par les enfants. Exemple : Le contributeur au niveau de l'abonnement s'applique à tous les groupes de ressources et ressources de cet abonnement.

Attribution des rôles — Lier un rôle à un principal de sécurité au niveau d'une portée. Créez une affectation pour accorder l'accès ; supprimez-le pour révoquer l’accès.

Azure RBAC est un modèle autorisé

Azure RBAC est basé sur l'autorisation : les affectations accordent des autorisations (lecture, écriture, suppression). Plusieurs affectations sont additives (par exemple, lire à partir d'une affectation + écrire à partir d'une autre = lire et écrire). Pas d'actions définir les opérations à exclure d'un rôle ; autorisations effectives = Actions moins NotActions. Par exemple, Contributor a * dans Actions (toutes les opérations du plan de contrôle) et utilise NotActions pour exclure : la suppression des attributions de rôles, la création d'attributions de rôles, l'octroi de l'accès utilisateur à l'administrateur au niveau du locataire et la création/suppression d'artefacts de plan. Le résultat est l’ensemble d’autorisations de contributeur effectif.

Examiner les services de domaine Microsoft Entra

Aperçu

De nombreuses applications métier s'exécutent sur des appareils appartenant à un domaine et utilisent ADDS pour l'authentification et Politique de groupe pour la gestion. Lors du déplacement de ces applications vers Azure, vous avez besoin d’options d’authentification. Les alternatives incluent : un VPN de site à site (le trafic d'authentification traverse le VPN) ou contrôleurs de domaine répliques en tant que machines virtuelles Azure (la réplication traverse le VPN ; l'authentification reste dans le cloud). Les deux ajoutent des coûts et de la gestion.

Services de domaine Microsoft Entra est une alternative gérée. Il fonctionne dans le cadre de Microsoft Entra ID P1 ou P2 et fournit des services de domaine— Politique de groupe , rejoindre un domaine , Kerberos —à votre locataire Entra. Il est compatible avec AD DS sur site, vous n'avez donc pas besoin de déployer ou de gérer des contrôleurs de domaine dans le cloud.

Avec Microsoft EntraConnect , les utilisateurs peuvent utiliser les mêmes informations d'identification organisationnelles dans AD DS sur site et dans Entra Domain Services. Vous pouvez également utiliser les services de domaine Entra comme cloud uniquement service (pas d'AD DS sur site) : créez un locataire Entra, activez les services de domaine Entra et utilisez un réseau virtuel pour que les ressources sur site puissent utiliser les services de domaine gérés.

Avantages

  • Pas besoin de gérer, mettre à jour ou surveiller les contrôleurs de domaine.
  • Pas besoin de déployer ou de gérer la réplication Active Directory.
  • Aucun groupe d'administrateurs de domaine ou d'administrateurs d'entreprise pour le domaine géré.

Limites

  • Seul le ordinateur de base L'objet Active Directory est pris en charge.
  • Schéma ne peut pas être prolongé pour le domaine Entra Domain Services.
  • La structure de l'UO est plate ; Les unités d'organisation imbriquées ne sont pas prises en charge.
  • Il y a un GPO intégré pour les comptes d'ordinateur et d'utilisateur.
  • GPO intégrés impossible de cibler les unités d'organisation . Les filtres WMI et le filtrage des groupes de sécurité ne sont pas pris en charge.

Cas d'utilisation

Migrer les applications qui utilisent LDAP , NTLM , ou Kerberos du sur site au cloud. Exécutez des applications telles que Serveur SQL ou Serveur SharePoint sur des machines virtuelles Azure sans contrôleurs de domaine dans le cloud ou VPN sur site.

Activation et tarification

Activez les services de domaine Microsoft Entra via le Portail Azure . La facturation est par heure en fonction de la taille du répertoire.

Abonnements et gouvernance

Cadre d'adoption du cloud et gouvernance du cloud

Le Cadre d'adoption du cloud Microsoft pour Azure fournit des conseils de bout en bout pour l’adoption du cloud. Gouvernance du cloud est la gestion de l'utilisation du cloud : conformité réglementaire, sécurité, opérations, coûts, données, gestion des ressources et IA. La méthodologie Govern comporte cinq étapes : (1) Construire une équipe de gouvernance ; (2) Évaluer les risques liés au cloud ; (3) Documenter les politiques de gouvernance du cloud ; (4) Imposer des politiques avec des outils automatisés et une surveillance manuelle ; (5) Moniteur utilisation et conformité. Répétez les étapes 2 à 5 au fil du temps. Considérations clés : risque commercial et tolérance, politique et conformité (risque → déclarations de politique) et processus (surveillance des violations). Cinq disciplines principales : Gestion des coûts , Base de référence en matière de sécurité , Cohérence des ressources , Référence d'identité , Accélération du déploiement .

Azure Policy comme principal outil de gouvernance

Politique Azure est le principal outil de gouvernance d’Azure. Il applique les normes organisationnelles et évalue la conformité à grande échelle sur les ressources actuelles et nouvelles. Avantages : gestion centralisée, tableau de bord de conformité agrégé, prévention des erreurs de configuration, correction groupée et automatique, intégration avec les pipelines Azure DevOps. Exemples : autoriser uniquement certaines régions ou tailles de VM ; appliquer les balises ; nécessiter une MFA ; exiger des journaux de diagnostic pour Log Analytics ; appliquer la géo-réplication. La politique évalue les ressources et peut prévenir des ressources non conformes ne soient pas créées ou remédier ressources existantes (par exemple ajouter des balises manquantes). Vous pouvez exempter des ressources spécifiques d'une stratégie. Concevoir une politique pour équilibrer le contrôle et la stabilité avec la vitesse et la productivité.

Hiérarchie pour la gouvernance

Quatre niveaux : Groupes de gestion → Abonnements → Groupes de ressources → Ressources . Les paramètres appliqués à un niveau s'appliquent à cette étendue et aux niveaux inférieurs ; les niveaux inférieurs héritent. La stratégie attribuée à un groupe d’administration s’applique à tous les abonnements (et à leurs groupes de ressources et ressources) dans ce groupe.

Azure Resource Manager et Azure Policy

Plan de contrôle — Gère les ressources dans l'abonnement. Azure Resource Manager (ARM) gère les opérations du plan de contrôle ; Azure Policy est intégré à ARM. Les requêtes (portail, PowerShell, CLI, REST, SDK) passent par ARM (auth, RBAC, puis Policy). Plan de données — Interaction directe avec les données de ressources (par exemple, blob de stockage, requête SQL, secret Key Vault). Le plan de données est géré par le fournisseur de ressources ; La politique peut imposer que les ressources soient conformes. Soutiens politiques Modes du fournisseur de ressources pour les politiques prenant en compte le plan de données (par ex. Microsoft.Kubernetes.Data , Microsoft.KeyVault.Data , Microsoft.Network.Data , Microsoft.ManagedHSM.Data , Microsoft.DataFactory.Data , Microsoft.MachineLearningServices.v2.Data ). Champ vert : la politique existe en premier ; La demande de création/mise à jour est évaluée par la stratégie (après RBAC). Friches industrielles : les ressources existent en premier ; la nouvelle stratégie est appliquée via une analyse de conformité (automatique toutes les 24 heures ou manuelle) az policy state trigger-scan ).

Ressources Azure Policy

Définitions — Décrire les conditions de conformité et l'effet lorsque la condition est remplie. Stocké dans un emplacement de définition (groupe de gestion ou abonnement) ; cet emplacement définit l'endroit où la politique/initiative peut être attribuée. Portée = groupes de gestion, abonnements, groupes de ressources, ressources.

Initiatives (ensembles de politiques) — Regroupez plusieurs définitions de stratégie en une seule unité attribuable. Simplifier l'affectation et la gestion ; suivre la conformité pour un objectif plus large (par exemple, le cadre réglementaire). Initiatives intégrées = collections des fournisseurs de ressources Azure (cadres de conformité, bonnes pratiques de sécurité). Initiatives personnalisées = politiques que vous définissez ; initiative personnalisée = groupe de définitions personnalisées. JSON utilisé pour la structure de définition d'initiative.

Missions — Attribuez une définition ou une initiative à une étendue (groupe de gestion, abonnement ou groupe de ressources). Facultatif : sélecteurs de ressources (déploiement progressif par emplacement/type), remplacements (effet de modification lors de l'affectation), Mode d'application (Désactivé = « et si » : évaluer mais n'appliquer pas l'effet), étendues exclues, messages de non-conformité, paramètres. Pour deployIfNotExists , attribuez une identité gérée (attribuée par le système ou par l'utilisateur) pour la correction.

Exonérations — Exempter une ressource ou une hiérarchie de l'évaluation. Les ressources exonérées comptent pour la conformité mais ne sont pas évaluées (renonciation temporaire). Catégories : Atténué (l'intention s'est rencontrée d'une autre manière), Renoncer (acceptation provisoire de non-conformité).

Attestations - Pour manuel -politiques d'effet : définir l'état de conformité des ressources/portées via des attestations ; une attestation par ressource et par affectation de stratégie manuelle.

Corrections — Mettre en conformité les ressources non conformes pour modify ou deployIfNotExists missions. La tâche de correction exécute le correctif ; correspondance de ressources nouvelles/mises à jour deployIfNotExists / modify peut être corrigé automatiquement.

Définitions Azure Policy (anatomie)

Politique = condition (si) + effet (alors). Écrit en JSON. Éléments clés : displayName , description , policyType (Intégré, Personnalisé, Statique), mode (par exemple, indexé pour ARM ; modes de fournisseur de ressources pour les politiques de plan de données), parameters , policyRule ( if + then ).

Créer des signatures d'accès partagé

UN signature d'accès partagé (SAS) est un URI qui accorde un accès restreint aux ressources de stockage sans exposer les clés de compte. Offrez un SAS aux clients pour un accès pendant une durée limitée. Types : Délégation des utilisateurs SAS — sécurisé avec les informations d'identification Microsoft Entra et les autorisations SAS ; pris en charge pour Blob et Data Lake. SAS au niveau du compte - accès à tout ce qu'un SAS de niveau service peut autoriser ainsi qu'à d'autres ressources (par exemple créer des systèmes de fichiers). Atténuation des risques : Utilisez toujours HTTPS pour la création et la distribution (évitez l’homme du milieu). Référencez les politiques d’accès stockées lorsque cela est possible. Déterminer les stratégies de réplication

Les données sont toujours répliquées pour des raisons de durabilité et de disponibilité. Possibilités :

LRS (stockage localement redondant) — Coût le plus bas ; moindre durabilité; trois copies dans un centre de données. À utiliser lorsque les données peuvent être reconstruites, sont transitoires ou lorsque la gouvernance nécessite une réplication à un emplacement unique.

ZRS (stockage redondant de zone) — Réplication synchrone sur trois clusters de stockage sur trois zones de disponibilité dans une région. Bonnes performances et faible latence ; données disponibles si une zone est en panne. Pas dans toutes les régions ; le passage à ZRS peut nécessiter un déplacement de données.

GRS (Stockage géo-redondant) — Réplique vers un région secondaire (à des centaines de kilomètres). Une plus grande durabilité ; 16 9 ; protège des pannes régionales. Lire à partir du secondaire uniquement si Microsoft lance le basculement. RA-GRS ajoute un accès en lecture à la région secondaire à tout moment.

GZRS (stockage géo-redondant) — ZRS dans la région primaire plus réplication vers la région secondaire. Combine la redondance de zone avec la protection régionale contre les catastrophes. 16 9. RA-GZRS ajoute un accès en lecture au secondaire. Microsoft recommande GZRS/RA-GZRS pour la cohérence, la durabilité, la haute disponibilité, les performances et la reprise après sinistre.

Ajouter des règles de gestion du cycle de vie des objets blob

Configurer la disponibilité des machines virtuelles

Les administrateurs Azure se préparent aux pannes planifiées et imprévues avec un plan de disponibilité qui comprend des stratégies de maintenance, des temps d'arrêt inattendus et des architectures de machines virtuelles résilientes.

Planifier la maintenance et les temps d'arrêt

Maintenance matérielle non planifiée : La plateforme Azure prédit que le matériel ou un composant de plateforme associé à une machine physique est sur le point de tomber en panne et émet un événement de maintenance matérielle imprévue. Azure utilise Migration en direct pour déplacer la VM d'un matériel défaillant vers un matériel sain avec seulement une courte pause ; les performances peuvent être réduites avant ou après l’événement.

Temps d'arrêt inattendu : Des pannes inattendues (par exemple, des pannes de réseau local, de disque local ou au niveau du rack) incitent Azure à réparer/migrer automatiquement la machine virtuelle vers un hôte sain dans le même centre de données. Pendant la réparation, les machines virtuelles subissent un temps d'arrêt (redémarrage) et peuvent perdre le lecteur temporaire dans certains cas.

Entretien prévu : Mises à jour périodiques de Microsoft sur la plateforme Azure sous-jacente pour améliorer la fiabilité, les performances et la sécurité. Azure peut nécessiter le redémarrage de l'hôte pour terminer les mises à jour ; cela ne corrige pas automatiquement le système d'exploitation ou le logiciel de votre VM.

Créer des groupes à haute disponibilité

Un ensemble de disponibilité est un regroupement logique de machines virtuelles associées afin qu'un point de défaillance unique n'affecte pas toutes les machines. Le regroupement permet de garantir que toutes les machines virtuelles ne sont pas mises à niveau en même temps lors des opérations hôtes planifiées.

Principales caractéristiques :

  • Toutes les machines virtuelles d’un groupe à haute disponibilité doivent exécuter le même ensemble de fonctionnalités et avoir le même logiciel installé.
  • Azure répartit les machines virtuelles de l'ensemble sur des serveurs physiques, des racks de calcul, des unités de stockage et des commutateurs réseau.
  • Si une panne matérielle ou logicielle Azure se produit, seul un sous-ensemble de machines virtuelles est affecté ; l'application reste disponible.
  • Une machine virtuelle peut être ajoutée à un groupe à haute disponibilité uniquement lorsque la machine virtuelle est créée. Pour modifier le groupe à haute disponibilité ultérieurement, vous devez supprimer et recréer la VM.
  • Les groupes à haute disponibilité ne protègent pas contre les pannes au niveau du système d'exploitation ou des applications : vous avez toujours besoin d'une reprise après sinistre et d'une sauvegarde au niveau des applications.

Examiner les domaines de mise à jour et les domaines de pannes

Les ensembles de disponibilité utilisent deux concepts de placement pour améliorer la haute disponibilité et la tolérance aux pannes : mettre à jour les domaines et domaines de pannes . Chaque VM est placée dans un domaine de mise à jour et un domaine de pannes.

Mettre à jour le domaine : Groupe de nœuds mis à niveau ensemble lors d'une mise à niveau/déploiement d'un service. Caractéristiques : un seul domaine de mise à jour est redémarré à la fois pendant la maintenance planifiée ; la valeur par défaut est cinq domaines de mise à jour (non configurables par l'utilisateur) ; vous pouvez en configurer jusqu'à 20.

Domaine de panne : Groupe de nœuds partageant une unité physique de défaillance commune (par exemple, des nœuds dans le même rack partageant des commutateurs d'alimentation ou de réseau). Deux domaines de pannes fonctionnent ensemble pour atténuer les pannes matérielles, les pannes de réseau, les interruptions de courant et les mises à jour logicielles.

Vérifier les zones de disponibilité

Zones de disponibilité protégez les applications et les données contre les pannes du centre de données en conservant les ressources dans des emplacements physiques distincts au sein d’une région Azure (avec alimentation, refroidissement et mise en réseau indépendants). Déployez le calcul, le stockage, la mise en réseau et les données dans une zone, et répliquez-les sur plusieurs zones.

Principales caractéristiques : Il existe généralement au moins trois zones dans les régions activées ; chaque zone est constituée d'un ou plusieurs centres de données ; la séparation des zones protège des pannes du centre de données.

Comparez la mise à l'échelle verticale et horizontale

Évolutivité augmente le débit de la VM proportionnellement aux ressources matérielles disponibles afin que le système puisse traiter davantage de requêtes sans impact inacceptable sur le temps de réponse/le débit.

Mise à l'échelle verticale (augmentation/réduction) : Changez la taille de la VM pour augmenter/diminuer le CPU/mémoire (VM plus ou moins puissante). Utile lorsqu'une VM est sous-utilisée (réduire pour réduire les coûts) ou lorsque vous avez besoin de plus de capacité sans ajouter de nouvelles VM.

Mise à l'échelle horizontale (scale-out/in) : Augmentez/diminuez le nombre d'instances de VM pour correspondre aux changements de charge de travail. Généralement plus flexible dans le cloud et peut prendre en charge une grande échelle, souvent avec moins de contraintes que l'évolution vers un matériel plus volumineux. Un reprovisionnement peut être nécessaire lors du redimensionnement ou de la modification du modèle de VM.

Implémenter des groupes identiques de machines virtuelles Azure

Ensembles de machines virtuelles identiques Azure déployer et gérer un ensemble de VM identiques. Lorsque vous configurez toutes les instances de la même manière, vous bénéficiez d’une véritable mise à l’échelle automatique : Azure augmente automatiquement les instances à mesure que la demande augmente et les diminue lorsque la demande diminue. Les groupes identiques permettent de créer des services à grande échelle sans pré-provisionner manuellement des centaines de machines virtuelles.

Principales caractéristiques :

  • Toutes les instances de VM sont créées à partir de la même image et configuration du système d'exploitation de base.
  • Les groupes identiques s'intègrent à Équilibreur de charge Azure (couche 4) et Passerelle d'applications Azure (couche 7, y compris la terminaison TLS/SSL).
  • Si une instance rencontre un problème, les clients peuvent continuer à accéder à l'application via d'autres instances.
  • Les groupes identiques prennent en charge la mise à l’échelle automatique et peuvent évoluer en fonction de la demande ou du calendrier.
  • Limites : jusqu'à 1 000 exemples ; si vous créez/téléchargez des images personnalisées, la limite est 600 .

Créer des groupes de machines virtuelles identiques

Dans le portail Azure, vous spécifiez le nombre d'instances de machine virtuelle et leurs tailles, ainsi que des préférences telles que l'utilisation d'instances Azure Spot, de disques managés Azure et de stratégies d'allocation.

Paramètres importants :

  • Mode orchestration : Flexible vous permet d’ajouter manuellement des machines virtuelles de n’importe quelle configuration au groupe identique ; Uniforme utilise un modèle de machine virtuelle pour qu'Azure génère des instances identiques.
  • Image: Choisissez l’image de base du système d’exploitation/de l’application.
  • Architecture de la VM : x64 (meilleure compatibilité) ou Bras64 (souvent un meilleur rapport qualité-prix).
  • Taille: Choisissez en fonction de la charge de travail ; cela détermine la capacité de calcul, de mémoire et de stockage et affecte le coût horaire.

Avancé: Algorithme de propagation contrôle la façon dont les machines virtuelles sont équilibrées entre les domaines de pannes. Épandage maximum répartit les machines virtuelles sur autant de domaines de pannes que possible dans chaque zone. Épandage fixe s'étend sur exactement cinq domaines de pannes ; si moins de cinq domaines de pannes sont disponibles, la propagation fixe peut échouer. Microsoft recommande Épandage maximum .

Implémenter la mise à l'échelle automatique

L'autoscaling augmente/diminue automatiquement les instances de VM en fonction de règles définies afin que vous puissiez répondre à l'évolution de la demande tout en minimisant la capacité inutile.

Points à considérer :

  • Capacité ajustée automatiquement : Créez des règles avec des seuils afin que les changements de capacité se produisent lorsque les métriques répondent aux conditions définies.
  • Évoluer : Lorsque la demande augmente de manière constante, ajoutez des instances pour maintenir les performances.
  • Mise à l'échelle : Lorsque la demande diminue régulièrement (par exemple, le soir/le week-end), supprimez des instances pour réduire les coûts.
  • Événements programmés : Augmentez/diminuez automatiquement la capacité à heures fixes.
  • Aérien: La mise à l'échelle automatique réduit les frais opérationnels liés à la surveillance et à l'optimisation.

Paramètres de mise à l'échelle (exemple) : Seuil de pourcentage d'utilisation du processeur ; nombre d'instances à mettre à l'échelle ; durée de la requête (fenêtre de rétrospection pour la stabilisation des mesures) ; calendrier (dates de début/fin et jours de répétition).

Sauvegarde Azure — Sauvegarde en tant que service pour les machines physiques et virtuelles (sur site et cloud). Scénarios : fichiers/dossiers sous Windows ; instantanés cohérents avec les applications (VSS) ; SQL Server, SharePoint, Exchange ; machines virtuelles Azure natives (Windows et Linux) ; Clients Windows 10 et Linux. Avantages : Gestion automatique du stockage (pay-as-you-use) ; mise à l'échelle illimitée ; LRS ou GRS ; transfert de données illimité (sans frais de transfert) ; chiffrement en transit et au repos ; points de récupération cohérents avec les applications ; rétention à long terme. Composants : Déployez le bon composant en fonction de ce que vous protégez : agent de sauvegarde Azure, System Center DPM, serveur de sauvegarde Azure ou extension de machine virtuelle de sauvegarde Azure. Les sauvegardes vont dans un Coffre des services de récupération (soutenu par Azure Storage) ; définir la politique de sauvegarde (planification, conservation).

Qu'est-ce qu'un modèle ARM ?

Modèles ARM sont Fichiers JSON qui définissent l’infrastructure et la configuration. Ils utilisent déclaratif syntaxe (indiquez ce que vous voulez), non impérative (commandes pas à pas). Vous déclarez des ressources et des propriétés ; Resource Manager les déploie dans l'ordre (en parallèle lorsque cela est possible).

Avantages des modèles ARM

  • Idempotent : Déployez le même modèle à plusieurs reprises et obtenez le même état de ressource.
  • Resource Manager orchestre la commande et valide le modèle avant le déploiement.
  • Pénétrer dans lié ou imbriqué modèles à réutiliser ; lien au moment du déploiement (les modèles liés peuvent utiliser un jeton SAS).
  • Historique de déploiement dans le portail Azure (paramètres et sorties).
  • Intégrez avec CI/CD (par exemple Azure Pipelines, GitHub Actions).

Structure du fichier de modèle ARM

Élément Description
schéma Requis. URL du schéma JSON pour le modèle.
version du contenu Requis. Version du modèle (par exemple 1.0.0.0).
Profil API Facultatif. Collection de versions d'API pour les types de ressources.
paramètres Facultatif. Valeurs fournies au moment du déploiement (fichier, CLI ou portail).
variables Facultatif. Valeurs pour simplifier les expressions.
fonctions Facultatif. Fonctions définies par l'utilisateur pour le modèle.
ressources Requis. Ressources à déployer ou à mettre à jour (groupe de ressources ou abonnement).
sorties Facultatif. Valeurs renvoyées après le déploiement.

Déployer un modèle ARM

  • Modèle local : az login → créer ou utiliser un groupe de ressources ( az group create --name <name> --location "<location>" ) → az deployment group create --name <deploymentName> --resource-group <rg> --template-file $templateFile . Préférer az deployment group create sur le obsolète az group deployment create .
  • Modèles liés : Le modèle principal déclenche le déploiement du modèle lié ; stockez le modèle lié et sécurisez-le avec un jeton SAS si nécessaire.
  • CI/CD : Azure Pipelines ou GitHub Actions pour le déploiement automatisé de modèles.

Ajout de ressources au modèle

Format du type de ressource : {fournisseur de ressources}/{type de ressource} (par ex. Microsoft.Storage/storageAccounts ). Ensemble type , apiVersion , name , location et spécifiques aux ressources properties . Utilisez la documentation Azure (par exemple « Définir les ressources dans les modèles ARM ») pour la version et les propriétés de l'API de chaque ressource.

Paramètres du modèle ARM

Les paramètres personnalisent le déploiement par environnement (dev, test, prod). Utilisez-les pour les valeurs qui changent : SKU, taille, capacité, noms de ressources. Max. 256 paramètres par modèle. Propriétés des paramètres : type , defaultValue , allowedValues , minValue / maxValue , minLength / maxLength , metadata.description .

Types : chaîne, secureString, entiers, booléen, objet, secureObject, tableau.

Bonnes pratiques : Décrivez chaque paramètre ; utilisez les valeurs par défaut lorsque cela est possible. Ne codez jamais en dur les noms d'utilisateur/mots de passe : utilisez les paramètres avec secureString (ou secureObject pour les objets sensibles). Les valeurs sécurisées ne sont pas lisibles après le déploiement.

Référence dans le modèle : [parameters('parameterName')] . Au moment du déploiement : --parameters storageAccountType=Standard_LRS (CLI) ou utilisez un fichier de paramètres.

Sorties du modèle ARM

Les sorties renvoient des valeurs après un déploiement réussi. Chaque sortie peut avoir : condition (booléen facultatif ; vrai par défaut), type , value (expression), copy (facultatif, pour plusieurs valeurs). Utiliser [reference('resourceName')] pour obtenir l'état d'exécution d'une ressource déployée (par ex. [reference('learntemplatestorage123').primaryEndpoints] ).

Redéploiement : Les modèles sont idempotents. Déployez à nouveau sans modification du modèle → aucune modification de l'environnement. Modifier un paramètre (ou un modèle) → seule cette modification est appliquée. Les ressources sont créées si elles sont manquantes, mises à jour uniquement lorsque quelque chose a changé.

Comparer les conteneurs aux machines virtuelles

La virtualisation matérielle vous permet d'exécuter plusieurs systèmes d'exploitation isolés sur le même matériel physique. Conteneurs représentent la prochaine étape de la virtualisation des ressources informatiques en virtualisant l'environnement du système d'exploitation afin que plusieurs applications puissent s'exécuter de manière isolée, tout en partageant le noyau du système d'exploitation hôte.

Comparer Conteneurs Machines virtuelles
Isolement Isolation légère de l'hôte et des autres conteneurs ; pas une frontière de sécurité aussi forte que les machines virtuelles. Isolation complète du système d'exploitation hôte et des autres machines virtuelles ; limite plus forte pour les charges de travail concurrentes.
Système opérateur Exécute la partie mode utilisateur d'un système d'exploitation ; adapté aux services nécessaires ; moins de ressources système. Exécute le système d'exploitation complet, y compris le noyau ; nécessite plus de CPU, de mémoire et de stockage.
Déploiement Déployer des conteneurs individuels avec Docker ; déployez des ensembles multi-conteneurs à l’aide d’un orchestrateur comme AKS. Déployez des machines virtuelles individuelles avec des outils tels que Windows Admin Center/Hyper-V Manager ; gérer de nombreuses machines virtuelles avec des automatisations/outils.
Stockage persistant Le stockage local peut utiliser des disques Azure (nœud unique) ; le stockage partagé peut utiliser Azure Files (partages SMB). Le stockage local utilise des disques VHD ; le stockage partagé peut utiliser les partages de fichiers SMB.
Tolérance aux pannes Si un nœud de cluster échoue, l'orchestrateur recrée les conteneurs sur un autre nœud. Les machines virtuelles peuvent basculer et redémarrer sur un autre serveur ; Le système d'exploitation redémarre sur le nouvel hôte.

Points à considérer lors de l'utilisation de conteneurs

  • Flexibilité et rapidité : Développez et partagez rapidement du code d’application conteneurisé.
  • Essai: Les builds basées sur des conteneurs simplifient les tests d'applications.
  • Déploiement d'application : Déploiements rationalisés et accélérés.
  • Densité de la charge de travail : Exécutez plus de charges de travail par hôte pour une meilleure utilisation des ressources.

Examiner les instances de conteneur Azure

Azure Container Instances (ACI) constitue le moyen le plus rapide et le plus simple d’exécuter un conteneur dans Azure. Il exécute des conteneurs sans gérer de machines virtuelles ni adopter un service d'orchestration de conteneurs de niveau supérieur. ACI est idéal pour les charges de travail pouvant s’exécuter dans des conteneurs isolés.

Ce qu’il faut savoir sur les instances de conteneur Azure

  • Temps de démarrage rapides : Les conteneurs démarrent en quelques secondes sans déployer de VM.
  • Connectivité IP publique et noms DNS : Exposez les conteneurs directement avec une adresse IP et un nom de domaine complet.
  • Tailles personnalisées : Les nœuds de conteneurs peuvent être mis à l'échelle dynamiquement pour répondre à la demande.
  • Stockage persistant : Montez les partages Azure Files dans des conteneurs.
  • Conteneurs Linux et Windows : Spécifiez le type de système d'exploitation lors de la création de groupes de conteneurs.
  • Groupes coprogrammés : Les groupes multi-conteneurs partagent les ressources de la machine hôte.
  • Déploiement du réseau virtuel : Déployez ACI dans un réseau virtuel Azure.

Comprendre les images de conteneurs

Les conteneurs sont créés à partir de images de conteneurs . Une image est un package exécutable léger et autonome qui comprend :

  • Code (source de l'application)
  • Durée d'exécution (environnement requis pour l'exécution)
  • Outils système
  • Bibliothèques système
  • Paramètres (configuration spécifique à l'application)

Lorsque vous exécutez une image, vous obtenez une image en cours d'exécution récipient (l'image au moment de l'exécution). Les images sont portables dans tous les environnements, ce qui permet aux conteneurs de fonctionner de manière cohérente.

Implémenter des groupes de conteneurs

La ressource de niveau supérieur dans ACI est le groupe de conteneurs . Un groupe de conteneurs est un ensemble de conteneurs planifiés sur la même machine hôte. Les conteneurs d'un groupe partagent un cycle de vie, un réseau local et des volumes de stockage.

Choses à savoir sur les groupes de conteneurs

  • Un groupe de conteneurs est similaire à un pod Kubernetes : un pod peut contenir plusieurs conteneurs ; les conteneurs peuvent partager des ressources associées.
  • ACI alloue des ressources pour un groupe multi-conteneurs en ajoutant les demandes de ressources de tous les conteneurs (CPU, mémoire, GPU).
  • Les groupes multi-conteneurs peuvent être déployés à l'aide de Modèles ARM ou Fichiers YAML (YAML est courant pour les déploiements de conteneurs uniquement).
  • Les groupes de conteneurs peuvent partager une adresse IP externe, des ports exposés et une étiquette DNS avec un nom de domaine complet.
  • Pour autoriser l'accès externe, vous devez exposer le port sur le conteneur et le groupe.
  • Le mappage de ports n'est pas pris en charge car les conteneurs d'un groupe partagent un espace de noms de port.
  • Lorsqu'un groupe de conteneurs est supprimé, son adresse IP et son nom de domaine complet sont libérés.

Exemple de configuration

Exemple de groupe multi-conteneurs (deux conteneurs) : le groupe s'exécute sur un seul hôte, possède une étiquette de nom DNS et expose une seule adresse IP publique avec un port exposé. Un conteneur écoute sur le port 80 (Web) et l'autre sur le port 1433. Le groupe comprend deux partages de fichiers Azure Files montés en tant que volumes afin que chaque conteneur puisse accéder à ses données.

Éléments à prendre en compte lors de l'utilisation de groupes de conteneurs

  • Mises à jour de l'application Web : Utilisez des groupes multi-conteneurs pour séparer les problèmes (par exemple, un conteneur sert l'application Web tandis qu'un autre extrait le contenu du contrôle de code source).
  • Collecte de données de journal : Un conteneur génère des journaux/métriques tandis qu'un autre les expédie vers un stockage à long terme.
  • Surveillance des applications : Un conteneur de surveillance demande périodiquement au conteneur d'applications de vérifier qu'il est en cours d'exécution ; alerte lorsque des problèmes sont détectés.
  • Front-end et back-end : Exécutez ensemble un conteneur frontal et un conteneur de services back-end dans un seul groupe.

Examiner les applications de conteneur Azure

Azure Container Apps (ACA) est une plateforme sans serveur permettant d'exécuter des applications conteneurisées. Il réduit les frais de gestion de l’infrastructure et de l’orchestration. Il est couramment utilisé pour :

  • Déploiement des points de terminaison de l'API
  • Hébergement de tâches de traitement en arrière-plan
  • Traitement piloté par les événements
  • Exécuter des microservices

Les applications conteneurisées peuvent évoluer en fonction du trafic HTTP, du traitement piloté par les événements, de la charge CPU/mémoire et de tout scaler pris en charge par KEDA, y compris la mise à l'échelle jusqu'à zéro.

Configurer Azure App Service

Implémenter les plans Azure App Service

Dans Azure App Service, une application s'exécute dans un Forfait App Service . Un plan App Service définit un ensemble de ressources de calcul (similaire à une « batterie de serveurs ») pour une ou plusieurs applications Web. Lorsque vous ajoutez des applications au même plan, elles partagent les mêmes ressources de calcul sous-jacentes dans ce plan.

Ce qu’il faut savoir sur les forfaits App Service

Lorsque vous créez un plan App Service dans une région, Azure crée des ressources de calcul pour ce plan dans la région spécifiée. Les applications placées dans le plan s'exécutent sur ces ressources de calcul.

Chaque plan App Service définit :

  • Système opérateur: Linux ou Windows.
  • Région: Où s’exécutent les ressources de calcul du plan.
  • Niveau tarifaire : Détermine les fonctionnalités d’App Service et le montant que vous payez. Les niveaux disponibles dépendent du système d'exploitation sélectionné au moment de la création.
  • Nombre d'instances de VM : Capacité d’échelle utilisée par les applications du plan.
  • Taille des instances de VM : Caractéristiques du processeur, de la mémoire et du stockage.

Vous pouvez ajouter de nouvelles applications à un forfait existant à condition que celui-ci dispose d'une capacité restante suffisante.

Éléments à prendre en compte lors de l’utilisation des plans App Service

  • Économies de coûts : Placez plusieurs applications dans le même plan pour partager des ressources de calcul.
  • Plusieurs applications dans un seul plan : Un forfait peut prendre en charge de nombreuses applications. Vous devez gérer soigneusement la capacité du plan, car les applications partagent les mêmes machines virtuelles.
  • Capacité du forfait : Avant d'ajouter une application, vérifiez que le plan dispose de suffisamment de capacité CPU/mémoire/stockage pour la nouvelle charge de travail.
  • Isolement des applications : Créez un nouveau plan lorsqu'une application est gourmande en ressources, nécessite une mise à l'échelle indépendante ou doit s'exécuter dans une région géographique différente.

Important: La surcharge d’un plan App Service peut entraîner des temps d’arrêt pour les applications nouvelles et existantes.

Déterminer le prix du plan Azure App Service

Le niveau tarifaire détermine les fonctionnalités App Service dont vous bénéficiez et le montant que vous payez. Exemples : Gratuit, Partagé, Basique, Standard, Premium, PremiumV2, PremiumV3, Isolé et IsoléV2.

La façon dont les applications fonctionnent et évoluent dépend du niveau du forfait :

  • Les applications du même plan partagent les mêmes instances de VM (exemple : si le plan utilise cinq instances, chaque application du plan s'exécute sur les cinq).
  • Si le plan est configuré pour la mise à l'échelle automatique, toutes les applications du plan évoluent ensemble en fonction des paramètres de mise à l'échelle automatique.

Les niveaux du forfait sont regroupés en trois catégories :

Catégorie Description
Calcul partagé Les niveaux Gratuit et Partagé exécutent des applications sur les mêmes machines virtuelles Azure que les autres clients. Les quotas de CPU sont alloués par application. Ces niveaux sont destinés au développement et aux tests ; les ressources ne peuvent pas évoluer.
Calcul dédié Basic, Standard, Premium, PremiumV2 et PremiumV3 exécutent des applications sur des machines virtuelles Azure dédiées. Seules les applications du même forfait partagent les mêmes ressources de calcul dédiées. Les niveaux supérieurs offrent généralement une plus grande capacité d'instance pour la mise à l'échelle.
Isolé Les niveaux Isolé et IsoléV2 exécutent des machines virtuelles dédiées sur des réseaux virtuels Azure dédiés. Cela fournit une isolation du réseau ainsi qu'un calcul dédié et prend en charge les capacités d'évolutivité les plus élevées.

Exemples par niveau :

  • Gratuit et partagé : Développement/test uniquement ; pas de SLA ; dosé par application ; plusieurs clients partagent les mêmes machines virtuelles.
  • Basique: Charges de travail à trafic réduit ; moins de fonctionnalités avancées ; équilibrage de charge réseau intégré ; prend en charge les conteneurs Linux (Web App for Containers).
  • Standard: Charges de travail de production ; équilibrage de charge intégré ; inclut la mise à l'échelle automatique ; prend en charge les conteneurs Linux.
  • Premium / Premium v2/v3 : Performances améliorées ; des processeurs plus rapides, un stockage SSD et un rapport mémoire/cœur accru (Premium v2). Prend en charge une échelle plus élevée via un nombre d’instances accru.
  • Isolé: Charges de travail critiques qui nécessitent le déploiement d’un réseau virtuel. Utilise un environnement App Service ; environnement dédié dans un centre de données Azure ; peut évoluer jusqu'à 100 instances.

Tâche à effectuer : Sélectionner un plan App Service

Dans le portail Azure, examinez les plans App Service disponibles et choisissez-en un en fonction des exigences matérielles et fonctionnelles (CPU, mémoire, instances de mise à l'échelle, sauvegardes, emplacements intermédiaires et redondance de zone). Astuce : tenez compte à la fois du matériel et des fonctionnalités lors de la sélection d’un forfait.

Augmenter et étendre Azure App Service

La mise à l’échelle d’Azure App Service utilise deux méthodes :

  • Mise à l'échelle : Augmentez le processeur, la mémoire et le stockage en modifiant le niveau tarifaire du plan App Service. Cela peut fournir des fonctionnalités supplémentaires telles que des machines virtuelles dédiées, des domaines/certificats personnalisés, des emplacements intermédiaires, une mise à l'échelle automatique, etc.
  • Évoluer : Augmentez le nombre d'instances de VM exécutant votre application. Les instances évolutives peuvent être configurées manuellement ou automatiquement (autoscale). Avec des niveaux isolés utilisant des environnements App Service, la mise à l’échelle peut atteindre jusqu’à 100 instances.

Mise à l'échelle automatique : Une fois configuré, la mise à l'échelle automatique ajuste automatiquement le nombre d'instances de mise à l'échelle horizontale en fonction des règles et des planifications.

Points à considérer :

  • Ajustez manuellement les niveaux du forfait : Commencez bas (par exemple, gratuit/partagé) et augmentez si nécessaire, puis réduisez lorsque les fonctionnalités ne sont plus nécessaires.
  • Mise à l'échelle automatique pour assister les utilisateurs et réduire les coûts : Continuez à servir pendant les périodes de pointe et réduisez la capacité lorsque la charge diminue.
  • Aucun redéploiement requis : La modification des paramètres de balance ne prend généralement que quelques secondes et s'applique à toutes les applications du plan sans modifier le code de l'application.
  • Mettre à l'échelle les dépendances séparément : La mise à l’échelle du plan App Service ne met pas automatiquement à l’échelle les services dépendants comme Azure SQL Database ou Azure Storage.

Configurer la mise à l'échelle automatique d'Azure App Service

Autoscale vous aide à exécuter la quantité appropriée de ressources pour gérer la charge des applications tout en réduisant les coûts en supprimant la capacité inutilisée. Pour utiliser la mise à l'échelle automatique, définissez :

  • Nombre d'instances minimum et maximum
  • Règles (déclencheurs + actions à grande échelle)

Notions clés :

  • Les paramètres de mise à l'échelle automatique sont regroupés en profils .
  • Les règles de mise à l'échelle automatique incluent un déclenchement (basé sur des métriques ou basé sur le temps) et un action à grande échelle (dedans ou dehors).
  • Lorsque les conditions de la règle sont remplies, une ou plusieurs actions d'échelle sont déclenchées.
  • Autoscale peut envoyer des notifications par e-mail et/ou webhooks.

Types de déclencheurs :

  • Basé sur des métriques : Échelle basée sur la charge mesurée (par exemple, utilisation du processeur au-dessus d'un seuil). Les mesures courantes incluent le temps CPU, le temps de réponse moyen et les requêtes.
  • Basé sur le temps (programmé) : Échelle à des heures prédéterminées (par exemple, à 8h00 le samedi dans un fuseau horaire).

Éléments à prendre en compte lors de la configuration de la mise à l'échelle automatique :

  • Nombre minimum d'instances : Assurez-vous que l'application est toujours en cours d'exécution même lorsqu'il n'y a aucune charge.
  • Nombre maximal d'instances : Plafonner les coûts en limitant l’utilisation horaire la plus élevée possible.
  • Marge d’échelle adéquate : Gardez min et max différents pour éviter une oscillation serrée.
  • Combinaisons de règles : Utilisez à la fois des règles de montée en charge et de mise à l'échelle (augmentation et diminution) pour éviter les pannes/problèmes de performances et éviter des coûts inutiles.
  • Statistiques métriques : Choisissez la statistique correcte (Moyenne, Minimum, Maximum, Total) pour la métrique.
  • Nombre d'instances par défaut : Choisissez une valeur par défaut sûre, car la mise à l'échelle automatique s'adapte à cette valeur lorsque les métriques ne sont pas disponibles.
  • Notifications : Configurez les notifications pour rester informé de la façon dont la charge modifie votre capacité.

Exemples de paramètres de mise à l'échelle : Seuil du processeur, nombre d'instances à adapter, durée de la requête (durée d'analyse), et un calendrier (jours de répétition et plage horaire).

Sécurisez votre application App Service

App Service fournit une authentification et une autorisation intégrées afin que vous puissiez sécuriser les applications Web, les applications API, les backends mobiles et les applications Azure Functions avec peu ou pas de code.

Comment App Service sécurise les applications :

Options de sécurité des applications :

  • Autoriser les requêtes anonymes : Différer l'autorisation à votre application ; App Service transmet les informations d’authentification à votre application pour les demandes authentifiées.
  • Autoriser uniquement les requêtes authentifiées : Rediriger les demandes anonymes vers le point de terminaison de connexion du fournisseur ; pour les applications mobiles natives, un HTTP 401 est renvoyé ; convient lorsque l'ensemble de l'application nécessite une authentification (pas idéal pour les applications avec des pages d'accueil publiques).
  • Journalisation et traçage : Afficher les traces d'authentification/autorisation dans les fichiers journaux (y compris les références à EasyAuthModule_32/64 dans le suivi des demandes ayant échoué).

Configurer les réseaux virtuels

Présentation des réseaux virtuels Azure

Azuré réseaux virtuels (VNets) constituent la base des réseaux privés dans Azure. Ils permettent aux ressources Azure de communiquer entre elles, avec Internet et avec les réseaux locaux en toute sécurité. Cas d'utilisation : créez un réseau dédié uniquement au cloud privé, étendez votre centre de données avec une connectivité sécurisée ou activez le cloud hybride (connectez les applications cloud aux systèmes sur site).

Objectifs d'apprentissage (depuis Configurer les réseaux virtuels ) : décrire les fonctionnalités et les composants d'Azure VNet ; identifier les fonctionnalités et les cas d'utilisation des sous-réseaux et des sous-réseaux ; identifier les cas d'utilisation des adresses IP privées et publiques ; créez un réseau virtuel et attribuez une adresse IP. Prérequis : connaissance de base des réseaux virtuels et familiarité avec les formats d'adresses IP et les sous-réseaux.

Planifier des réseaux virtuels

Choses à savoir : Un réseau virtuel Azure est un isolement logique des ressources cloud Azure. Vous pouvez provisionner et gérer VPN dans Azur. Chaque réseau virtuel a son propre CIDR bloc et peut être lié à d’autres réseaux virtuels et réseaux sur site (hybrides ou inter-site) lorsque les blocs CIDR ne se chevauchent pas . Vous contrôlez Paramètres du serveur DNS et segmentation de sous-réseau pour le réseau virtuel.

Scénario Description
Réseau virtuel dédié au cloud privé uniquement Pas de interlocalisation. Les services et les machines virtuelles du réseau virtuel communiquent directement et en toute sécurité dans le cloud. Configurez les points de terminaison pour les ressources nécessitant Internet.
Étendre le centre de données VPN site à site avec IPsec entre votre passerelle VPN et Azure pour faire évoluer la capacité du centre de données.

Créer des sous-réseaux

Sous-réseaux fournir des divisions logiques au sein d’un réseau virtuel pour la sécurité, les performances et la gestion. Conditions : chaque sous-réseau dispose d'une plage d'adresses IP dans l'espace d'adressage du réseau virtuel ; la portée doit être unique et sans chevauchement avec d'autres sous-réseaux ; utiliser CIDR notation. Vous pouvez segmenter un réseau virtuel en un ou plusieurs sous-réseaux dans le portail Azure.

Adresses réservées : Réserves d'Azur cinq adresses IP par sous-réseau. Pour 192.168.1.0/24 : .0 = adresse réseau ; .1 = passerelle par défaut ; .2 et .3 = Azure DNS ; .255 = diffusion. Ceux-ci ne peuvent pas être affectés à des ressources.

Points à considérer : (1) Exigences du service — Certains services nécessitent un sous-réseau dédié (par ex. Passerelle VPN Azure nécessite un sous-réseau de passerelle) ; assurez-vous de disposer de suffisamment d’espace non alloué. (2) Appliances virtuelles réseau (NVA) — Par défaut, Azure achemine le trafic entre les sous-réseaux ; vous pouvez remplacer le routage via une NVA en plaçant des ressources dans différents sous-réseaux. (3) Groupes de sécurité réseau (NSG) — Zéro ou un NSG par sous-réseau ; les règles autorisent ou refusent le trafic. (4) Lien privé — Connectivité privée du réseau virtuel vers Azure PaaS, les services clients ou partenaires ; évite l’exposition publique sur Internet.

Créer des réseaux virtuels

Lors de la création d'un réseau virtuel : définissez le Espace d'adressage IP (utilisez une plage qui n'est pas déjà utilisée ; peut être sur site ou dans le cloud, pas les deux) ; définir au moins un sous-réseau ; chaque plage de sous-réseau doit être unique et ne se chevauche pas au sein du réseau virtuel. Dans le portail Azure, indiquez l’abonnement, le groupe de ressources, le nom du réseau virtuel et la région. Consultez la documentation sur la mise en réseau Azure pour connaître les limites par défaut actuelles.

Planifier l'adressage IP

Les ressources Azure utilisent des adresses IP pour communiquer avec d'autres ressources Azure, des réseaux locaux et Internet. Deux types : privé (au sein du réseau virtuel et sur site via VPN ou ExpressRoute) et publique (Internet et services publics Azure). Les adresses IP peuvent être statiquement ou dynamiquement attribué ; vous pouvez utiliser différents sous-réseaux pour statique ou dynamique. Utiliser du statique quand : la résolution DNS nécessiterait des mises à jour des enregistrements d’hôte ; Règles de sécurité ou de pare-feu basées sur IP ; Certificats TLS/SSL liés à IP ; machines virtuelles basées sur des rôles (par exemple, contrôleurs de domaine, serveurs DNS).

Créer et associer des adresses IP publiques

Créez des adresses IP publiques dans le portail Azure (par exemple pour une VM ou un équilibreur de charge). Paramètres: Version IP — IPv4 peut s'attacher à la carte réseau ou à l'équilibreur de charge ; IPv6 uniquement pour les équilibreurs de charge ; UGS — doit correspondre au SKU de l'équilibreur de charge (les adresses IP publiques du SKU de base ont été retirées le 30 septembre 2025 ; utilisez Standard) ; Étage — faire correspondre l'équilibreur de charge (régional ou inter-région) ; Affectation — Statique (attribué à la création, libéré lors de la suppression de la ressource). Les adresses IP publiques sont souvent utilisées avec des équilibreurs de charge.

Configurer le peering de réseau virtuel Azure

Peering de réseaux virtuels Azure connecte des réseaux virtuels dans la même région ou dans des régions différentes et fournit une communication sécurisée et privée entre les ressources. Utilisez-le lorsque des réseaux virtuels distincts doivent communiquer sans exposer les services à Internet. Objectifs d'apprentissage : Identifier les cas d'utilisation et les fonctionnalités ; configurer la passerelle VPN pour le transit ; étendez le peering avec des routes en étoile, définies par l'utilisateur (UDR) et le chaînage de services. Prérequis : mise en réseau cloud et réseaux virtuels de base ; familiarité avec les outils de test de connectivité.

Types : peering régional — Connecte les réseaux virtuels dans la même région Azure (même région de cloud public, ou même Chine, ou même région Azure Government). Appairage mondial — Connecte les réseaux virtuels dans différentes régions (toutes régions publiques ou chinoises ; le peering mondial entre différentes régions Azure Government n'est pas autorisé). Les réseaux virtuels peuvent être appairés entre les abonnements et les locataires. Après le peering, les deux réseaux virtuels se comportent comme un seul pour la connectivité ; ils restent des ressources distinctes à gérer.

Avantage Description
Connexions réseau privées Le trafic entre les réseaux virtuels appairés reste sur le backbone Microsoft ; aucun Internet public, passerelle ou cryptage n'est requis.
Forte performance Connexion à faible latence et à bande passante élevée sur l'infrastructure Azure.
Communication simplifiée Les ressources d’un réseau virtuel peuvent communiquer avec les ressources du réseau virtuel appairé.
Transfert de données fluide Appairage entre les abonnements, les modèles de déploiement et les régions.
Aucune interruption des ressources Aucun temps d'arrêt lors de la création ou après le peering.

Passerelle de transit : Lorsque les réseaux virtuels sont appairés, vous pouvez utiliser un Passerelle VPN dans un réseau virtuel (par exemple, un hub) comme point de transit. L'autre réseau virtuel (en rayons) peut utiliser la passerelle distante pour atteindre un réseau local (VPN de site à site), un autre réseau virtuel (de réseau virtuel à réseau virtuel) ou des clients (de point à site). Le transit de passerelle est pris en charge pour le peering régional et mondial. Un réseau virtuel ne peut avoir qu’une seule passerelle VPN ; avec le transit par passerelle, les réseaux virtuels satellites n'ont pas besoin de leur propre passerelle. Dans le portail, configurez le peering avec des options pour autoriser/transférer le trafic (transit de passerelle). Les NSG peuvent toujours autoriser ou bloquer l’accès entre des réseaux virtuels ou des sous-réseaux appairés.

Créer du peering : Utilisez le portail Azure, PowerShell ou CLI. Vous avez besoin du Contributeur au réseau (ou un rôle personnalisé avec des autorisations d'appairage) et deux réseaux virtuels (le second est le réseau distant ). Créez le peering du premier VNet au second ; initialement, les machines virtuelles ne peuvent pas communiquer tant que le peering n'est pas établi. Statut d'appairage : Dans le portail, vérifiez l'état de la connectivité. Le peering ne réussit que lorsque les deux Afficher les réseaux virtuels Connecté . Pour Resource Manager : le statut est Initié lorsque vous créez le peering initial à partir du premier réseau virtuel, alors Connecté lorsque le côté distant accepte. Le déploiement classique utilise également Mise à jour .

Étendre le peering : non transitif, hub-and-spoke, UDR, chaînage de services : Le peering est non transitif . Si A–B et B–C sont appairés, A et C n'obtiennent pas automatiquement la connectivité ; seuls les réseaux directement peerés peuvent communiquer. Pour étendre la connectivité et le transit, utilisez : (1) En étoile — Hub VNet héberge une passerelle NVA ou VPN ; tous les rayons sont connectés au hub ; le trafic circule à travers le hub. (2) Itinéraires définis par l'utilisateur (UDR) — Le saut suivant dans un UDR peut être l'adresse IP d'une VM dans un réseau virtuel appairé ou une passerelle VPN. (3) Chaînage de services — Dirigez le trafic d'un réseau virtuel vers une NVA ou une passerelle en configurant des UDR dont le prochain tronçon est une VM dans un réseau virtuel appairé ou une passerelle de réseau virtuel. Cela permet au trafic de circuler via les appareils centraux du hub.

Des routes système supplémentaires sont créées lorsque vous activez peering , chaînage de services , un passerelle de réseau virtuel , ou points de terminaison de service .

Itinéraires personnalisés — itinéraires définis par l'utilisateur (UDR) : Remplacez les routes du système pour envoyer le trafic via des pare-feu ou des NVA. Créer un table de routage , ajoutez des routes, associez la table à un ou plusieurs sous-réseaux (un sous-réseau possède au plus une table de routage). Types de tronçon suivant pour les UDR :

Appliance virtuelle réseau (NVA) : Une VM qui fait office de pare-feu, d'optimiseur WAN, de contrôleur de distribution d'applications, de routeur, d'équilibreur de charge, d'IDS/IPS ou de proxy. Déployez depuis Azure Marketplace (par exemple Cisco, Check Point, Barracuda) ou utilisez une machine virtuelle Linux/Windows avec transfert IP. Placez le NVA dans un sous-réseau de périmètre (DMZ) ou utiliser microsegmentation (sous-réseaux de pare-feu dédiés ; tout le trafic acheminé via NVA pour inspection au niveau de la couche 4 ou 7). Activer Transfert IP sur la carte réseau Azure de la NVA ( az network nic update --ip-forwarding true ) et à l'intérieur du système d'exploitation (par exemple Linux sysctl -w net.ipv4.ip_forward=1 ) afin que le trafic non destiné à la NVA soit transféré. Certaines NVA utilisent plusieurs cartes réseau (gestion + trafic). Planifier pour haute disponibilité donc l’échec de la NVA n’interrompt pas la connectivité.

Exemple : acheminer le trafic via NVA : Créer une table de routage ; ajouter une route pour le sous-réseau privé (par exemple 10.0.1.0/24) avec le type de saut suivant Appliance virtuelle et IP du prochain saut = IP privée de NVA (par exemple 10.0.2.4). Associez la table de routage au publique sous-réseau. Déployez la NVA dans le sous-réseau DMZ (par exemple dmzsubnet 10.0.2.0/24) et activez le transfert IP sur sa carte réseau et dans le système d'exploitation. Résultat : le trafic du sous-réseau public vers le sous-réseau privé passe par la NVA ; le trafic dans la direction opposée peut utiliser les routes par défaut (directes), sauf si vous ajoutez également un UDR sur le sous-réseau privé. Vérifiez avec traceroute (public → privé devrait afficher NVA comme premier saut).

Configurer les groupes de sécurité réseau

Introduction aux groupes de sécurité réseau

Groupes de sécurité réseau (NSG) limiter le trafic réseau aux ressources de votre réseau virtuel. Un NSG contient règles de sécurité qui autorisent ou refusent le trafic entrant ou sortant. Utilisez-les pour sécuriser à la fois la mise en réseau des machines virtuelles et la mise en réseau des services Azure et pour empêcher le trafic indésirable ou non sécurisé d’atteindre les systèmes clés.

Objectifs d'apprentissage : Déterminer quand utiliser les NSG ; créer des NSG ; mettre en œuvre et évaluer les règles NSG ; décrire la fonction de groupes de sécurité des applications (ASG) . Prérequis : familiarité avec les réseaux virtuels et les machines virtuelles Azure ; connaissance pratique du portail Azure ; compréhension de base de l’acheminement et du contrôle du trafic.

Implémenter des groupes de sécurité réseau

Vous pouvez attribuer un NSG à un sous-réseau ou à un interface réseau (NIC) . Un NSG peut être associé plusieurs fois (à plusieurs sous-réseaux ou cartes réseau). Vous créez le NSG et définissez les règles de sécurité dans le portail Azure ; la page Présentation de la VM affiche les NSG associés, les sous-réseaux/cartes réseau attribués et les règles.

NSG et sous-réseaux : Attribuez un NSG à un sous-réseau pour créer un sous-réseau filtré (DMZ : tampon entre le réseau virtuel et Internet). Le NSG restreint le trafic à toutes les ressources de ce sous-réseau. Chaque sous-réseau peut avoir au plus un NSG.

NSG et interfaces réseau : Attribuez un NSG à une carte réseau pour contrôler tout le trafic via cette interface. Chaque carte réseau peut avoir zéro ou un NSG. Une carte réseau dans un sous-réseau peut avoir son propre NSG en plus du NSG du sous-réseau.

Déterminer les règles du groupe de sécurité réseau

Les règles de sécurité filtrent le trafic entrant et sortant des sous-réseaux et des cartes réseau. Azure crée règles par défaut dans chaque NSG (par ex. RefuserToutEntrant , AutoriserInternetSortant ). Vous ne pouvez pas supprimer les règles par défaut ; vous pouvez les remplacer en ajoutant une règle avec un priorité plus élevée (numéro de priorité inférieure).

Conditions de règle :

Paramètre Valeurs
Source N'importe lequel, adresses IP, mon adresse IP, numéro de service ou groupe de sécurité d'application
Plages de ports sources Ports autorisés ou refusés par la règle
Destination N'importe lequel, adresses IP, numéro de service ou groupe de sécurité d'application
Protocole TCP, UDP, ICMP ou Any (par défaut)
Action Autoriser ou refuser
Priorité 100 à 4 096 ; unique au sein du NSG ; valeur inférieure = priorité plus élevée

Règles entrantes par défaut refuser tous les flux entrants, sauf ceux provenant du réseau virtuel et de l’équilibreur de charge Azure. Règles sortantes par défaut autoriser le trafic vers Internet et le réseau virtuel. Ajoutez des règles personnalisées pour autoriser ou refuser un trafic spécifique ; traiter les règles par ordre de priorité.

Déterminer des règles de sécurité efficaces

Les NSG sont évalués de manière indépendante. Pour entrant trafic : Azure évalue le sous-réseau NSG d'abord, puis le Carte réseau NSG. Pour sortant : Carte réseau NSG d'abord, puis sous-réseau NSG. Le trafic intra-sous-réseau (VM dans le même sous-réseau) est également soumis à ces règles.

Points à considérer : (1) Autoriser tout le trafic — Si vous n'associez pas de NSG à un sous-réseau ou à une carte réseau, tout le trafic est autorisé selon les paramètres par défaut d'Azure. (2) Autoriser les règles aux deux niveaux — Pour autoriser le trafic, vous devez disposer d'une règle d'autorisation au niveau du sous-réseau et de la carte réseau si les deux disposent de NSG ; sinon, le trafic peut être refusé à un niveau. (3) Trafic intra-sous-réseau — Les règles NSG de sous-réseau s'appliquent au trafic entre les machines virtuelles du même sous-réseau ; vous pouvez refuser tout le trafic intra-sous-réseau avec une règle de refus. (4) Priorité des règles — Utiliser la valeur de priorité la plus basse possible pour les règles qui doivent être prioritaires ; laissez des espaces (par exemple 100, 200, 300) pour insérer de nouvelles règles sans renuméroter. Utilisez le Des règles de sécurité efficaces lien dans le portail Azure (ou Observateur de réseau ) pour voir quelles règles s'appliquent à une VM, un sous-réseau ou une carte réseau.

Créer des règles de groupe de sécurité réseau

Ajoutez des règles dans le portail Azure sous le NSG. Configurer Source (entrant : plage IP, ASG ou numéro de service), Destination (sortant : mêmes options), Service (protocole et plage de ports, par exemple HTTPS, RDP, SSH, FTP, DNS ou personnalisé), et Priorité . Les règles sont traitées par ordre de priorité sur le sous-réseau et les NSG de la carte réseau.

Implémenter des groupes de sécurité des applications

Groupes de sécurité des applications (ASG) vous permet de regrouper les machines virtuelles par charge de travail (par exemple, serveurs Web, serveurs d'applications). Vous attribuez la carte réseau d'une machine virtuelle à un ASG, puis utilisez l'ASG comme source ou destination dans les règles NSG au lieu des adresses IP. Cela donne une vue centrée sur l’application et simplifie la gestion des règles.

Exemple de scénario (niveaux Web + application) : Créer des ASG pour chaque niveau ; attribuez la carte réseau de chaque VM au bon ASG. Règles NSG : Priorité 100 — Autoriser Internet aux serveurs Web sur HTTP (80) et HTTPS (443) ; Priorité 110 — Autoriser les serveurs Web à accéder aux serveurs d'applications sur SQL (1433) ; Priorité 120 — Tout refuser aux serveurs d'applications sur HTTP/HTTPS. Résultat : seuls les serveurs Web peuvent atteindre le niveau application/base de données ; les bases de données sont protégées de l’accès direct à Internet.

Avantages des ASG : Pas besoin de conserver des adresses IP spécifiques dans les règles : ajoutez/supprimez des machines virtuelles de l'ASG et les règles s'appliquent automatiquement. Vous n'avez pas besoin d'aligner les machines virtuelles sur des sous-réseaux spécifiques ; regrouper par application et objectif. Moins de règles : une règle par ASG au lieu de par VM. L’organisation basée sur la charge de travail est plus facile à maintenir. Balises de service représenter les préfixes IP pour les services Azure et simplifier les règles pour le trafic des services Azure ; ASG regroupez vos machines virtuelles et vous permet de gérer la stratégie par groupe d’applications.

Résolution de noms et équilibrage de charge

Hébergez votre domaine sur Azure DNS – Introduction

DNS Azure héberge les enregistrements DNS pour vos domaines sur l’infrastructure Azure. Vous utilisez les mêmes informations d’identification, API, outils et facturation que les autres services Azure. Cas d'utilisation : vous disposez d'un domaine personnalisé provenant d'un registraire (par exemple Wideworldimporters.com) et avez besoin d'un service d'hébergement pour résoudre le domaine en IP de votre serveur Web. Objectif d'apprentissage : Configurez Azure DNS pour héberger votre domaine (et ajoutez des alias et d'autres enregistrements). Prérequis : connaissance de la résolution de noms et des adresses IP.

Qu’est-ce que DNS et Azure DNS ?

DNS (système de noms de domaine) est un protocole TCP/IP qui traduit les noms lisibles par l'homme (par exemple www.wideworldimports.com) en adresses IP. Les serveurs DNS soit (1) maintiennent un cache des recherches récentes et transmettre les requêtes inconnues à d'autres serveurs, ou (2) maintenir un faisant autorité base de données (paires clé-valeur) pour les domaines qu'ils desservent. Les appareils obtiennent les paramètres DNS du serveur sur site ou du FAI lorsqu'ils sont externes. Processus de recherche : vérifiez le cache → s'il n'est pas trouvé, interrogez d'autres serveurs DNS → mettez à jour le cache et résolvez, ou renvoyez « domaine introuvable ». IPv4 (par exemple 127.0.0.1) et IPv6 (par exemple fe80:11a1:ac15:…) sont tous deux pris en charge ; Le DNS peut résoudre l'un ou l'autre.

Types d'enregistrements DNS (courants) : UN — enregistrement de l'hôte, mappe le nom à l'adresse IPv4 ; AAAA - IPv6 ; CNAME — alias d'un nom à un autre (par exemple www vers le même site) ; MX — échange de courrier; SMS — texte (par exemple, vérification de domaine pour Azure/Microsoft 365). Autres : jokers, CAA, N.-É. (serveur de noms), SOA (début d'autorité), SPF, SRV. Azure DNS crée SOA et N.-É. automatiquement lorsque vous créez une zone. Ensembles d'enregistrements autoriser plusieurs valeurs dans un seul enregistrement (par exemple, un enregistrement A avec deux adresses IP) ; SOA et CNAME ne prennent pas en charge les jeux d'enregistrements.

DNS Azure vous permet d'héberger et de gérer des domaines sur une infrastructure de serveur de noms distribuée à l'échelle mondiale à l'aide de vos informations d'identification Azure. Azure DNS agit comme SOA pour le domaine. Vous ne pouvez pas enregistrer un domaine avec Azure DNS : utilisez un registraire tiers. Pourquoi l'utiliser : Construit sur Resource Manager (sécurité, facilité d'utilisation, zones DNS privées, enregistrements d'alias). Azure DNS ne prend pas en charge DNSSEC ; utilisez un hébergeur tiers si nécessaire. Sécurité: RBAC, journaux d'activité, verrouillage des ressources. Facilité d'utilisation : Mêmes informations d'identification et facturation ; gérer via le portail, PowerShell, CLI, REST API, SDK.

Zones DNS privées et enregistrements d'alias

Zones DNS privées — Non visible sur Internet ; aucun registraire n'est nécessaire. Utilisez-les pour résoudre les noms DNS des machines virtuelles dans vos réseaux virtuels et entre réseaux virtuels. Vous spécifiez quels réseaux virtuels peuvent résoudre les enregistrements dans la zone. Avantages : fait partie de l'infrastructure Azure (pas de DNS personnalisé) ; tous les types d'enregistrements (A, CNAME, TXT, MX, SOA, AAAA, PTR, SRV) ; noms d'hôtes automatiques pour les machines virtuelles ; DNS à horizon partagé (même nom dans les zones privées et publiques, résolu par origine de la demande).

Jeux d'enregistrements d'alias — Pointez vers une ressource Azure (par exemple, adresse IP publique, profil Traffic Manager, point de terminaison CDN, profil de porte d'entrée). Types d'enregistrement pris en charge : UN , AAAA , CNAME . Avantages : suivi du cycle de vie (le DNS se met à jour lorsque la ressource cible change) ; évitez les dossiers qui traînent ; prise en charge des applications à charge équilibrée au niveau sommet de la zone (par exemple Wideworldimporters.com pointant vers un équilibreur de charge).

Configurer Azure DNS pour héberger votre domaine

Zone DNS publique : (1) Créer une zone DNS dans Azure (abonnement, groupe de ressources, nom de zone, par exemple Wideworldimporters.com, emplacement). (2) Obtenez des serveurs de noms Azure DNS à partir de l'enregistrement NS (quatre serveurs de noms). (3) Mettre à jour le registraire de domaine — définissez les enregistrements NS du domaine sur les quatre serveurs de noms Azure DNS ( délégation ). (4) Vérifier la délégation - par ex. nslookup -type=SOA wideworldimporters.com ; peut prendre plus de 10 minutes. (5) Ajouter des enregistrements — Un enregistrement : Nom (par exemple serveur web1), Type A, TTL, adresse IP. CNAME : par exemple. nom www , tapez CNAME, TTL — à utiliser lorsque plusieurs noms (par exemple www et apex) doivent correspondre au même site.

Zone DNS privée : (1) Créer un zone DNS privée (groupe de ressources, nom de zone, par exemple private.wideworldimporters.com). (2) Identifiez les réseaux virtuels qui nécessitent une résolution de nom. (3) Créer liens de réseau virtuel — dans la zone, ajoutez un lien pour chaque VNet ; seuls les réseaux virtuels liés peuvent résoudre la zone privée.

Enregistrements de sommet et d'alias de zone

Le domaine sommet (apex de zone, sommet de racine) est le niveau supérieur du domaine (par exemple Wideworldimporters.com). Le @ le symbole le représente souvent dans les enregistrements DNS. CNAME n'est pas pris en charge au sommet de la zone pour Traffic Manager ou CDN ; enregistrements d'alias sont soutenus au sommet. Les enregistrements d'alias permettent au sommet de référencer les ressources Azure (Traffic Manager, CDN, adresse IP publique, porte d'entrée) et de se mettre à jour automatiquement lorsque l'adresse IP de la cible change. Vous pouvez ainsi pointer le sommet vers un équilibreur de charge sans suspendre les enregistrements lorsque l'adresse IP LB change. Créez un alias : ajoutez un jeu d'enregistrements avec un nom vide (apex), tapez A (ou AAAA/CNAME), Jeu d'enregistrements d'alias = Oui , Type d'alias = Ressource Azure, puis sélectionnez la ressource (par exemple, IP publique ou équilibreur de charge). Vérifiez avec nslookup ou en ouvrant l'adresse IP de la ressource dans un navigateur.

Introduction à l'équilibreur de charge Azure

Équilibreur de charge Azure distribue le trafic entrant sur un ensemble de machines virtuelles (ou d'instances d'ensemble de machines virtuelles identiques) dans un pool back-end . Vous configurez la distribution avec règles d'équilibrage de charge et évitez d'envoyer du trafic vers des nœuds défaillants avec sondes de santé . Il fonctionne à Couche 4 (couche de transport OSI), comme les équilibreurs de charge matériels traditionnels, adaptés à la migration d'appareils de couche 4 locaux vers Azure. Objectifs d'apprentissage : Découvrez ce qu'est Load Balancer et ce qu'il fournit ; déterminez si cela répond à vos besoins. Pré-requis : concepts de base en matière de réseautage.

Qu’est-ce qu’Azure Load Balancer ?

L'équilibrage de charge répartit le trafic entrant sur plusieurs serveurs afin qu'un seul serveur ne soit pas submergé. Azure Load Balancer distribue le trafic sur les machines virtuelles Azure ou les instances de groupes identiques et offre une haute disponibilité et des performances via des règles et des sondes d’intégrité.

Équilibreur de charge public — Bilans de charge trafic Internet à vos VM. Mappe les paramètres de l'équilibreur de charge IP et port publics au IP et port privés de machines virtuelles back-end. Peut également fournir sortant connexions pour les machines virtuelles dans le réseau virtuel. À utiliser pour les niveaux Web face à Internet.

Équilibreur de charge interne (privé) — Utilise un adresse IP privée à l'avant ; jamais exposé directement à Internet. Distribue le trafic vers les ressources à l'intérieur du réseau virtuel ou accessibles via VPN/ExpressRoute. Utiliser pour : équilibrage de charge au sein d'un réseau virtuel ; entre locaux (sur site sur les machines virtuelles du réseau virtuel) ; applications multi-niveaux (du niveau connecté à Internet aux niveaux back-end non connectés à Internet) ; Applications métiers dans Azure (y compris les serveurs sur site dans le pool). Les équilibreurs de charge publics et internes prennent en charge les scénarios entrants et sortants et s'adaptent à des millions de flux TCP et UDP.

Comment fonctionne Azure Load Balancer

Load Balancer fonctionne à Couche 4 (TCP/UDP ; IP et port source/destination, protocole). Composants clés :

  • IP frontale — Adresse à laquelle les clients se connectent ; peut être public ou privé. IP publique → équilibreur de charge public ; IP privée → équilibreur de charge interne. Plusieurs IP frontales prises en charge (par exemple pour plusieurs règles).
  • Règles d'équilibrage de charge — Mappez une adresse IP et un port front-end à un ensemble d’adresses IP et de ports back-end. Le trafic est distribué à l'aide d'un hachage à cinq tuples : IP source, port source, IP destination, port destination, protocole. Affinité de session (persistance) conserve le même client sur la même instance back-end. Vous pouvez équilibrer la charge de plusieurs ports ou IP ; pour une utilisation de routage basé sur le contenu (couche 7) Passerelle d'applications Azure .
  • Pool back-end — Groupe de machines virtuelles ou d’instances de groupes identiques qui reçoivent du trafic. L'ajout ou la suppression d'instances déclenche une reconfiguration automatique.
  • Sondes de santé — Déterminez si une instance est saine. Les instances défectueuses cessent de recevoir nouveau relations; les connexions existantes se poursuivent jusqu'à ce que l'application mette fin au flux, que le délai d'inactivité expire ou que la VM s'arrête. Types de sondes : TCP (session TCP réussie pour sonder le port), HTTP , HTTPS (par exemple HTTP 200 dans le délai d'attente). Configurez le port, l'intervalle et le seuil de mauvais fonctionnement.
  • Persistance de la session — Aucun (par défaut) : toute VM saine ; IP client (2 tuples) : même IP client → même instance ; IP client et protocole (3 tuples) : même IP client et même protocole → même instance.
  • Ports haute disponibilité (HA) — Règle avec protocole = Tout et port = 0 sur un plan interne Standard équilibreur de charge. La charge équilibre tous les flux TCP et UDP sur tous les ports ; décision par flux (cinq tuples). Utile pour les NVA et lorsque de nombreux ports doivent être équilibrés en charge.
  • Règles NAT entrantes — Transférer le trafic du front-end de l'équilibreur de charge vers le port d'une VM spécifique (par exemple, port IP public → TCP 3389 sur une VM pour RDP).
  • Règles sortantes — Configurer SNAT (Source NAT) afin que les instances back-end puissent atteindre Internet ou d'autres points de terminaison publics.

Quand utiliser et quand ne pas utiliser Azure Load Balancer

Utilisez Load Balancer lorsque : Vous avez besoin latence ultra faible et haute performance à la couche 4 ; vous remplacez le matériel de couche 4 sur site ; vous souhaitez que les sondes d'intégrité évitent les nœuds défaillants et la persistance de la session afin qu'un client s'en tienne à une VM par session. Utiliser publique des équilibreurs de charge pour le trafic Web provenant d'Internet ; interne équilibreurs de charge entre le niveau Web et le niveau intermédiaire (par exemple, analyse des données). Utiliser règles NAT entrantes pour RDP/SSH sur des machines virtuelles spécifiques.

N'utilisez pas Load Balancer lorsque : Une seule VM gère un faible trafic et n’a pas besoin de pool back-end.

Alternatives :

Service Cas d'utilisation
Porte d'entrée azur Équilibrage de charge global et accélération du site ; Couche 7 (déchargement TLS, routage basé sur le chemin, WAF, mise en cache) ; applications Web multirégionales.
Gestionnaire de trafic Azure Équilibrage de charge basé sur DNS ; répartition optimale dans les régions du monde ; basculement plus lent en raison de la mise en cache DNS/TTL.
Passerelle d'applications Azure ADC en tant que service ; Couche 7 (déchargement TLS, routage URL) ; régional (et non mondial).

Équilibreur de charge Azure est un système haute performance à latence ultra faible Couche 4 (TCP/UDP, entrant et sortant), conçu pour des millions de requêtes par seconde et une haute disponibilité redondante entre zones. C'est le cas pas fournir une fonctionnalité de pare-feu d'application Web (WAF) : utilisez Front Door ou Application Gateway pour cela.

Introduction à Azure Application Gateway

Passerelle d'applications Azure traite le trafic vers les applications Web hébergées sur un pool de serveurs Web (machines virtuelles, groupes identiques, App Service ou sur site). Il fournit Couche 7 équilibrage de charge du trafic HTTP, pare-feu d'applications Web (WAF) inspection, et TLS/SSL cryptage entre les clients et la passerelle et entre la passerelle et les serveurs back-end. Utilisez-le pour remplacer le matériel de périmètre qui effectue des contrôles de santé, la terminaison TLS, l'affinité de session et le filtrage de sécurité (par exemple, injection SQL, scripts intersites). Objectifs d'apprentissage : Découvrez ce qu'est Application Gateway et ce qu'elle fournit ; déterminez si cela répond à vos besoins. Prérequis : réseautage de base ; familiarité avec les machines virtuelles, App Service et les réseaux virtuels.

Qu’est-ce qu’Azure Application Gateway ?

Application Gateway gère les demandes des clients vers les applications Web sur un pool principal. Il équilibre la charge du trafic HTTP et peut inspecter le trafic avec WAF. Il prend en charge TLS/SSL pour la passerelle client et la passerelle back-end. Équilibrage de charge utilise un tournoi à la ronde processus; viscosité de la session garantit que les requêtes au cours de la même session sont envoyées au même serveur principal (important pour le commerce électronique et l'état de la session).

Caractéristiques: HTTP, HTTPS, HTTP/2 et WebSocket ; pare-feu d'applications Web pour se protéger contre les vulnérabilités ; chiffrement des requêtes de bout en bout ; mise à l'échelle automatique ; vidange de raccordement pour la suppression gracieuse des membres back-end lors des mises à jour.

Comment fonctionne Azure Application Gateway

Composants :

  • IP frontale — Une IP publique et/ou une IP privée. Les clients envoient des demandes à cette adresse.
  • Auditeurs — Acceptez le trafic sur une combinaison de protocole, de port, d'hôte et d'IP. Basique itinéraires d'écoute par chemin d'URL ; Multi-sites l'écouteur utilise également le nom d'hôte. Les écouteurs gèrent les certificats TLS/SSL (client-passerelle).
  • Règles de routage — Liez un auditeur aux pools back-end. Les règles utilisent le nom d'hôte et le chemin pour diriger les requêtes vers le bon pool. Associé Paramètres HTTP définir : le protocole vers le back-end, la persistance de la session, l'épuisement de la connexion, le délai d'expiration de la demande, les sondes d'intégrité. TLS/SSL facultatif vers le back-end (certificat ou géré par App Service).
  • Pools back-end — Machines virtuelles, groupe identique, App Service ou serveurs sur site. Même configuration et sécurité par pool. La passerelle peut recrypter le trafic vers le back-end à l'aide d'un certificat ; Les back-ends App Service sont fiables et chiffrés automatiquement.

Équilibrage de charge est un tournoi à la ronde à Couche 7 (Hôte et chemin URL), contrairement à Load Balancer (Couche 4, IP/port). La persistance de la session peut être configurée.

Pare-feu d'applications Web (WAF) - Facultatif; court devant l'auditeur. Vérifie les demandes par rapport OWASP règles (par exemple, injection SQL, scripts intersites, injection de commandes, contrebande/fractionnement HTTP, robots, violations de protocole). Utilisations Ensemble de règles de base (CRS) 3.2, 3.1, 3.0 ou 2.2.9 (3.1 par défaut). Vous pouvez activer des règles spécifiques ou personnaliser les éléments à inspecter et les limites de taille des messages.

Méthodes de routage : Routage basé sur le chemin — Différents chemins d'URL vers différents pools (par ex. /video/* → pool vidéo, /images/* → pool d'images). Routage multi-sites — Plusieurs noms d'hôte (par exemple contoso.com, fabrikam.com) sur la même passerelle ; chaque auditeur est dirigé vers un pool différent. Aussi: redirection (par exemple HTTP → HTTPS), Réécriture de l'en-tête HTTP , pages d'erreur personnalisées .

Terminaison TLS/SSL — Terminez TLS au niveau de la passerelle pour décharger le processeur des serveurs back-end ; pas de certificats sur les serveurs en cas de résiliation uniquement. Pour chiffrement de bout en bout , la passerelle décrypte avec sa clé puis rechiffre avec la clé publique du back-end. Seuls les ports 80/443 sont exposés sur la passerelle vers Internet, réduisant ainsi la surface d'attaque.

Sondes de santé — Déterminez quels serveurs back-end sont disponibles. Statut de réponse HTTP 200 à 399 = sain. La sonde par défaut attend 30 secondes avant de marquer comme indisponible. Mise à l'échelle automatique ajuste la capacité en fonction du trafic. WebSocket et HTTP/2 sont pris en charge nativement sur les ports 80 et 443.

Quand utiliser et quand ne pas utiliser Azure Application Gateway

Utilisez Application Gateway lorsque : Vous devez diriger le trafic vers un pool back-end (Azure ou sur site) ; des sondes de santé afin que le trafic ne soit pas envoyé vers des serveurs indisponibles ; Terminaison TLS pour réduire le processeur du serveur ; WAF pour bloquer l'injection SQL, XSS et autres menaces Web ; affinité de session pour les applications qui stockent l'état de la session sur le serveur.

Ne pas utiliser lorsque : Une seule application Web à faible trafic gère déjà la charge sans pool.

Comparaison: Porte d'entrée — Global Layer 7, WAF, mise en cache, multi-région. Gestionnaire de trafic — basé sur le DNS, global ; basculement plus lent en raison du DNS. Équilibreur de charge — Couche 4, régionale, latence ultra-faible. Passerelle d'applications — Régional, couche 7 (routage basé sur le chemin/l'hôte, WAF, déchargement TLS), adapté lorsque vous avez besoin d'un routage au niveau de l'application et d'un WAF dans une région.

Surveillez vos machines virtuelles Azure avec Azure Monitor

Surveillance des machines virtuelles Azure

Azure Monitor est une solution de surveillance complète permettant de collecter, d'analyser et de répondre à la télémétrie des ressources Azure et non Azure, y compris les machines virtuelles. Azure Monitor propose deux principaux types de données de surveillance : Métrique et Journaux .

Métrique sont des valeurs numériques collectées à intervalles réguliers (par exemple, CPU, utilisation des ressources, nombre d'erreurs, temps de réponse). Azure collecte automatiquement un ensemble prédéfini de métriques d'hôte de machine virtuelle et conserve les données pour 93 jours à quelques exceptions près.

Journaux sont des événements système horodatés qui incluent des données structurées ou de forme libre. Azure enregistre automatiquement les journaux d'activité pour les ressources Azure, tandis que Journaux Azure Monitor collecte des journaux supplémentaires dans un Espace de travail Log Analytics pour l'interrogation et l'analyse.

Couches de surveillance des machines virtuelles

La surveillance des VM s'étend généralement sur ces couches :

  • VM hôte : les ressources de calcul, de stockage et de réseau de l’hôte VM.
  • Système d'exploitation invité : charges de travail du système d’exploitation et du client VM.
  • Charges de travail et applications client : applications exécutées dans la VM.

Surveillance de la VM hôte

Métriques de l'hôte de VM mesurer des aspects techniques tels que l'utilisation du processeur et le fonctionnement de la machine. Utilisez ces métriques pour déclencher des alertes, identifier les tendances et gérer les coûts (par exemple, redimensionner les machines virtuelles en fonction des modèles d'utilisation).

Sur la page Présentation de la machine virtuelle, Azure fournit des graphiques intégrés pour les métriques d’hôte courantes :

Sauvegarde et récupération

Qu'est-ce que la Sauvegarde Azure ?

Azure Backup fournit des solutions simples, sécurisées et économiques pour sauvegarder les données et les récupérer à partir d'Azure. Il est conçu pour protéger les charges de travail de l’entreprise contre la perte de données tout en limitant les efforts opérationnels.

Définition de la sauvegarde Azure

Azure Backup est un service Azure qui fournit des solutions de sauvegarde rentables, sécurisées et sans infrastructure pour les ressources de données gérées par Azure. La gestion centralisée facilite la définition de politiques de sauvegarde et protège un large éventail de charges de travail, notamment :

  • Machines virtuelles (VM) Azure et disques managés Azure
  • Partages Azure Files
  • Bases de données SQL Server et SAP HANA exécutées sur des machines virtuelles Azure
  • Objets blob azur
  • Azure Database pour PostgreSQL (et offres flexibles/autres)
  • Azure Database pour MySQL (serveurs flexibles)
  • Clusters Azure Kubernetes (prise en charge au niveau de la charge de travail)
  • Fichiers, dossiers et état du système sur site

Quand utiliser la sauvegarde Azure

Utilisez Azure Backup lorsque vous avez besoin d’une gestion centralisée des sauvegardes d’entreprise dans l’ensemble de votre organisation et que vous souhaitez une sauvegarde/restauration en libre-service pour gérer des problèmes tels que des scénarios de corruption ou d’administrateur malveillant. Il s’agit également d’un outil idéal pour répondre aux exigences de conformité grâce à des politiques de rétention cohérentes.

Principales caractéristiques

Fonctionnalité Description Usage
Solution de sauvegarde sans infrastructure Aucun serveur de sauvegarde ni infrastructure supplémentaire à gérer ; Azure Backup gère et met à l'échelle automatiquement le stockage de sauvegarde. Élimine les dépenses en capital et réduit les frais généraux d’exploitation.
Gestion à grande échelle Gérer le patrimoine de sauvegarde depuis Centre de sauvegarde et automatisez à l'aide des API, PowerShell et Azure CLI. Vous aide à découvrir, gouverner, surveiller, exploiter et optimiser la gestion des sauvegardes en un seul endroit.
Sécurité Chiffrement intégré et modèles de connectivité sécurisés (par exemple, points de terminaison privés) ainsi que des alertes. Les sauvegardes sont protégées contre les ransomwares, les administrateurs malveillants et les suppressions accidentelles.

Comment fonctionnent l’objectif de temps de récupération (RTO) et l’objectif de point de récupération (RPO) ?

RTO est le délai cible dans lequel un processus métier doit être restauré après un sinistre (par exemple, tolérer jusqu'à 4 heures d'arrêt => RTO = 4 heures).

RPO est la quantité maximale de perte de données mesurée dans le temps (par exemple, sauvegarde toutes les heures => RPO = 1 heure). Lorsque vous définissez le RTO sur 3 heures, votre objectif est de restaurer l'accès dans les 3 heures afin de réduire l'impact opérationnel.

Comment fonctionne la sauvegarde Azure

Azure Backup protège les charges de travail en combinant une couche d'intégration de charge de travail (extensions de sauvegarde), des niveaux de stockage de plan de données, des contrôles de sécurité/disponibilité et un plan de gestion centralisé (coffres et centre de sauvegarde).

Quelles données sont sauvegardées et comment ?

Au niveau le plus simple, Azure Backup sauvegarde les données, l'état des machines et les charges de travail des machines locales et des instances de machines virtuelles Azure vers le stockage cloud Azure. Les données sauvegardées sont stockées dans Coffres Recovery Services et Coffres de sauvegarde .

Approches courantes :

  • Machines Windows sur site : utilisez l’agent Azure Backup MARS pour sauvegarder directement sur Azure, ou sauvegardez sur un serveur de sauvegarde tel que DPM ou MABS, puis sauvegardez ce serveur dans un coffre-fort Recovery Services.
  • Machines virtuelles Azure : Azure Backup installe un extension de sauvegarde sur la VM pour sauvegarder la VM ; si vous n'avez besoin que de fichiers/dossiers, utilisez la sauvegarde de fichiers de style MARS pour ces scénarios.

Types de sauvegarde pris en charge

Azure Backup prend en charge les sauvegardes complètes et les sauvegardes incrémentielles. La première sauvegarde est une sauvegarde complète ; les sauvegardes suivantes capturent uniquement les blocs de données modifiés depuis la sauvegarde précédente (incrémentielle).

Pour SQL Server, Azure Backup prend en charge les types de sauvegarde suivants :

Taper Description Usage
Complet Sauvegarde l'intégralité de la base de données et inclut suffisamment de journaux pour récupérer ces données. Déclenchez jusqu'à une sauvegarde complète par jour (peut être quotidienne ou hebdomadaire).
Différentiel Basé sur la sauvegarde complète la plus récente ; capture les données modifiées depuis cette sauvegarde complète. Déclenchez jusqu’à une sauvegarde différentielle par jour. Impossible de faire fonctionner le plein et le différentiel le même jour.
Plusieurs sauvegardes par jour Sauvegardes horaires pour les machines virtuelles Azure avec planification améliorée. RPO minimum de 4 heures et RPO maximum de 24 heures ; La politique de sauvegarde améliorée prend en charge les planifications de 4/6/8/12/24 heures.
Sauvegarde de disque sélective Sauvegardez et restaurez un sous-ensemble de disques de données connectés (pour une restauration instantanée et une restauration au niveau du coffre-fort). Utilisez la politique de sauvegarde améliorée pour sauvegarder de manière sélective les disques critiques (et éventuellement sauvegarder le disque du système d'exploitation uniquement pour réduire les coûts).
Journal des transactions Permet une restauration ponctuelle jusqu'à une seconde spécifique. Configurez les sauvegardes des journaux transactionnels toutes les 15 minutes (max).

Couche d'intégration de charge de travail - Extension de sauvegarde

Une extension de sauvegarde est installée par type de charge de travail sur la VM source ou une VM de travail. Au moment de la sauvegarde (telle que définie par la politique de sauvegarde), l'extension génère des données de sauvegarde, telles que :

  • Stockage: instantanés lors de la sauvegarde de machines virtuelles Azure ou d'Azure Files
  • Sauvegarde du flux : sauvegardes basées sur flux pour des bases de données telles que SQL Server ou SAP HANA

Les données de sauvegarde sont transférées vers le stockage géré Azure Backup dans le plan de données via des réseaux Azure sécurisés (à l'aide de modèles tels que des NSG, des pare-feu et des points de terminaison privés).

Plan de données – Niveaux d'accès

Azure Backup stocke les données dans trois niveaux d'accès :

  • Niveau d'instantané : Stockage d'instantanés spécifique à la charge de travail utilisé dans la première phase d'une sauvegarde de VM. Les restaurations sont plus rapides car les instantanés sont conservés localement dans l'abonnement du client dans un groupe de ressources spécifié.
  • Niveau standard du coffre-fort : stockage en ligne dans le stockage locataire géré par Microsoft (stockage de sauvegarde à mise à l'échelle automatique). Ajoute une couche de protection supplémentaire ; les données de sauvegarde restent disponibles même si la source de données d'origine est supprimée ou compromise.
  • Niveau d'archivage : données de sauvegarde à conservation à long terme (LTR) stockées pour des besoins de conformité, avec un accès moins fréquent.

Tous les niveaux ont des caractéristiques et des tarifs RTO différents. Certaines charges de travail prises en charge par le niveau d'instantané conservent une copie dans les niveaux d'instantané et standard du coffre-fort.

Plan de données - Disponibilité et sécurité

Les données de sauvegarde sont répliquées dans des zones ou des régions en fonction de la redondance que vous choisissez (par exemple, LRS , GRS , ou ZRS ). Azure protège les sauvegardes en chiffrant les données et en utilisant Azure RBAC pour contrôler qui peut effectuer des opérations de sauvegarde et de restauration.

Protection contre la suppression malveillante : Azure Backup inclut la suppression logicielle pour les sauvegardes. Si une sauvegarde est supprimée, elle est conservée pendant 14 jours (sans frais) afin que vous puissiez la récupérer si nécessaire.

Plan de gestion – Coffre Recovery Services/Coffre-fort de sauvegarde et Centre de sauvegarde

Azure Backup utilise les coffres Recovery Services et les coffres de sauvegarde pour orchestrer et gérer les sauvegardes et pour stocker les données de sauvegarde. Le coffre-fort fournit l'interface permettant de gérer les politiques de sauvegarde, les déclencheurs et la conservation.

Centre de sauvegarde offre une interface unique pour gérer les sauvegardes sur plusieurs types de charges de travail, coffres-forts, abonnements, régions et locataires Azure Lighthouse.

Quand utiliser Azure Backup (critères de décision)

Utilisez Azure Backup lorsque vous avez besoin de solutions de sauvegarde sécurisées et sans infrastructure pour les ressources de données gérées par Azure. Les principaux critères d'évaluation comprennent :

Critères Considération

Il fournit également une surveillance intégrée des tâches pour les opérations liées à la sauvegarde étendues au coffre-fort. Pour les grands parcs, Backup Explorer offre une visibilité opérationnelle agrégée sur l’ensemble du parc de sauvegarde, prenant en charge l’exploration et le dépannage.

Qu'est-ce que la Sauvegarde Azure ?

Azure Backup est un service Azure intégré qui fournit des sauvegardes sécurisées pour les ressources de données gérées par Azure à l'aide de solutions sans infrastructure pour les sauvegardes et les restaurations en libre-service. Il prend en charge la gestion à grande échelle à un coût inférieur et prévisible, avec des solutions de sauvegarde spécialisées pour les machines virtuelles Azure (et des sauvegardes tenant compte de la charge de travail pour SQL Server ou SAP HANA exécutés sur des machines virtuelles Azure).

Par rapport aux solutions de sauvegarde traditionnelles qui nécessitent une configuration importante, Azure Backup est facilement géré via le portail Azure.

Sauvegarde Azure et Azure Site Recovery

Azure Backup et Azure Site Recovery améliorent la résilience, mais ils le font différemment. La sauvegarde conserve des copies des données avec état afin que vous puissiez restaurer « dans le temps ». Site Recovery réplique les données presque en temps réel pour permettre le basculement.

Utilisez Site Recovery pour les catastrophes à l’échelle régionale (telles que les catastrophes naturelles). Utilisez la sauvegarde en cas de perte accidentelle de données, de corruption de données et de récupération de ransomware.

La meilleure approche dépend de la criticité de l’application, des exigences RPO/RTO et des implications financières.

Sauvegarder une machine virtuelle Azure à l'aide d'Azure Backup

Les machines virtuelles Azure sont sauvegardées en prenant des instantanés des disques sous-jacents à des intervalles définis par l'utilisateur et en transférant ces instantanés vers le coffre Recovery Services conformément à la stratégie définie par le client.

Coffre des services de récupération

Azure Backup utilise un Coffre des services de récupération pour gérer et stocker les données de sauvegarde. Un coffre-fort est une entité de gestion du stockage qui fournit une interface simple pour effectuer et surveiller les opérations de sauvegarde/restauration. Avec Azure Backup, vous spécifiez généralement le coffre-fort de la machine virtuelle que vous souhaitez protéger ; les données de sauvegarde sont transférées en arrière-plan vers les comptes de stockage Azure Backup gérés par le service.

Instantanés

UN instantané est une sauvegarde ponctuelle de tous les disques de la VM. Azure Backup utilise différentes extensions d'instantanés par système d'exploitation :

Extension Système d'exploitation Description
Instantané de la machine virtuelle Fenêtres Utilise le service VSS (Volume Shadow Copy Service) pour effectuer une copie des données sur le disque et en mémoire.
Instantané de machine virtuelle Linux Linux Crée une copie du disque pour l'instantané.

Niveaux de cohérence des instantanés

Application cohérente : capture la machine virtuelle dans son ensemble (utilise des rédacteurs VSS sous Windows et des scripts pré/post personnalisés sous Linux pour capturer l'état de l'application). Assure la cohérence de la VM et de toutes les applications en cours d'exécution.

Système de fichiers cohérent : créé même si VSS échoue sous Windows ou si les scripts pré/post échouent sous Linux. Empêche la corruption de la machine, mais les applications installées peuvent nécessiter un nettoyage au démarrage.

Crash cohérent : se produit généralement si la machine virtuelle est arrêtée au moment de la sauvegarde ; aucune opération d'E/S ni le contenu de la mémoire n'est capturé, cela ne garantit donc pas la cohérence du système d'exploitation/de l'application.

Politique de sauvegarde

La politique de sauvegarde définit la fréquence de sauvegarde et la durée de conservation. Les sauvegardes de VM peuvent être déclenchées quotidiennement ou hebdomadairement et conservées pendant plusieurs années. Les stratégies prennent en charge le niveau d’instantané et le niveau de coffre-fort. Avec la politique améliorée, vous pouvez déclencher des sauvegardes horaires. La politique de sauvegarde améliorée prend également en charge Sauvegarde de disque sélective pour sauvegarder uniquement les disques sélectionnés et restaurer un sous-ensemble à partir d'un point de récupération (utile pour les données critiques et l'optimisation des coûts).

Niveau d'instantané et restauration instantanée

Le niveau Instantané stocke les instantanés localement pendant une période maximale de cinq jours. Pour les récupérations opérationnelles, la restauration à partir d'instantanés est recommandée car elle est plus rapide : c'est ce qu'on appelle restauration instantanée .

Niveau du coffre-fort

Le niveau Vault transfère en outre les instantanés vers le coffre-fort pour plus de sécurité et une conservation plus longue. Le type de point de récupération devient instantané et coffre-fort .

Processus de sauvegarde pour une machine virtuelle Azure

À un niveau élevé, Azure Backup :

  • Démarre une tâche de sauvegarde pour les machines virtuelles Azure sélectionnées en fonction de la fréquence de sauvegarde dans la stratégie de sauvegarde.
  • Lors de la première sauvegarde, installe l'extension de snapshot de VM appropriée si la VM est en cours d'exécution (VM Snapshot pour Windows, VM SnapshotLinux pour Linux).
  • Prend l'instantané et le stocke initialement localement, puis le transfère vers le coffre-fort.
  • Optimise les sauvegardes en traitant les disques de VM en parallèle.
  • Ne transfère que les blocs modifiés (delta) depuis la sauvegarde précédente.
  • La copie complète des données d'instantané dans le coffre-fort peut prendre plusieurs heures pendant les heures de pointe ; la durée totale de sauvegarde des politiques quotidiennes est inférieure à 24 heures.

Options de protection supplémentaires : vous pouvez activer le chiffrement du coffre-fort avec des clés gérées par le client (CMK). La suppression logicielle améliorée pour un coffre-fort Recovery Services protège contre la suppression ; vous pouvez toujours conserver la suppression logicielle améliorée pour réduire les risques de suppression accidentelle ou d'attaques de logiciels malveillants.

Restaurer les données de la machine virtuelle

Pour tester les plans BCDR, Azure Backup propose des options de restauration. Vous pouvez restaurer à partir du niveau d'instantané (restauration instantanée ; généralement idéale pour la récupération opérationnelle) ou à partir du niveau de coffre-fort .

Types de restauration

Option de restauration Détails
Créer une nouvelle VM Crée une VM de base à partir du point de restauration. La nouvelle VM doit se trouver dans la même région que la VM source.
Restaurer le disque Restaure un disque de VM, puis crée une nouvelle VM (ou attache le disque à une VM existante). Azure Backup fournit un modèle pour personnaliser la machine virtuelle.
Remplacer l'existant Restaurez un disque et remplacez un disque sur la VM existante après avoir d'abord pris un instantané de la VM existante. Non pris en charge pour certains scénarios de restauration gérée dans ce module.
Région transversale (région secondaire) Restaurez les machines virtuelles/disques Azure dans une région secondaire couplée. Utilisé pour les tests de reprise après sinistre.
Restauration d'abonnements croisés Restaurez les machines virtuelles/disques gérés vers un abonnement différent au sein du même locataire (lorsque activé sur le coffre-fort). Fonctionne avec la restauration entre régions/zones et avec des identités de système géré (MSI). Non pris en charge pour les points de récupération de niveau instantanés.
Restauration interzone Restaurez une VM épinglée à une zone vers une autre zone (zone logique). Pris en charge pour les machines virtuelles gérées avec MSI et nécessite des paramètres de redondance de coffre/stockage compatibles.
Sauvegarde de disque sélective Sauvegardez et restaurez les disques de machine virtuelle sélectionnés à l'aide d'une stratégie améliorée (restauration d'un sous-ensemble de disques à partir d'un point de récupération).

Récupérer des fichiers à partir d'une sauvegarde : vous pouvez récupérer des fichiers individuels en montant l'instantané sur une machine cible à l'aide de l'initiateur iSCSI.

Restaurer une machine virtuelle chiffrée

Azure Backup prend en charge la sauvegarde et la restauration des machines virtuelles chiffrées avec Azure Disk Encryption. Disk Encryption s'intègre à Azure Key Vault pour gérer les secrets. Vous pouvez également utiliser des clés de chiffrement Key Vault (KEK) pour chiffrer les secrets avant qu’ils ne soient écrits dans Key Vault.

Limites: seul le chiffrement par clé autonome est pris en charge (les clés de partie certificat ne sont pas prises en charge). Les restaurations au niveau des fichiers/dossiers ne sont pas prises en charge : restaurez l'intégralité de la VM, puis copiez manuellement les fichiers/dossiers. L'option « Remplacer l'existant » n'est pas disponible pour les machines virtuelles chiffrées.
