1. Introduction to DevOps

Introduction to DevOps — Microsoft Learn (~42 min · 15 units · Beginner).

Learning objectives

  • Define DevOps and core principles (people, process, products for continuous value).
  • Plan organizational transformation: teams, shared goals, timelines, agile structure.
  • Describe Azure DevOps and GitHub at a high level.
  • Explain source control and Git fundamentals, working with Git locally.
  • Consider license management strategy.
  • Apply cycle time thinking and validated learning (data-informed decisions).

What is DevOps?

DevOps combines Development and Operations into multidisciplinary teams with shared practices, tools, and accountability. Typical practices include agile planning, continuous integration, continuous delivery, and monitoring. DevOps is a continuous journey, not a one-time project.

Business outcomes organizations often target: higher deployment frequency, shorter lead time, lower mean time to recovery (MTTR), and lower change failure rate—leading to faster time-to-market, fewer deployment incidents, better developer productivity, and lower cost through automation.

Cycle time, OODA, and validated learning

The OODA loop (Observe, Orient, Decide, Act) maps well to product delivery: observe metrics and telemetry, orient with experiments, decide priorities, act by shipping and measuring. Cycle time is how long it takes to go from idea or commit to production feedback—e.g. from code commit to deploy, or from feature request to customer signal. Long cycle time on small changes indicates a bottleneck.

Be data-informed, not paralyzed by data: some releases help, some hurt, some are neutral—fail fast on what does not advance goals and double down on what does (“pivot or persevere”). Validated learning uses factual, actionable, timely feedback (usage, performance, support, KPIs) each cycle.

Shorter cycle time comes from smaller batches, more automation, stronger pipelines, better telemetry, and more frequent deploys—which increases experiments and validated learning per unit of time.

GitHub Projects and boards

GitHub Projects help teams organize and visualize work (tables, boards, roadmaps). Project boards track issues and pull requests through customizable columns and automation. You configure projects, views (e.g. by assignee, milestone, iteration), and workflows so the team sees the same plan in one place.

Azure Boards

Azure Boards is Azure DevOps work tracking for Agile, Scrum, Kanban, and custom processes. It supports process customization (work item types, fields, workflows), hierarchy from tasks up to epics, queries (flat, tree, direct links), dashboards, and Delivery Plans for multi-team timelines and dependencies. Typical work items include Epic, Feature, User Story, Task, and Bug—with discussions, history, links to commits/PRs, attachments, and tags.

Link GitHub to Azure Boards

Integrating GitHub with Azure Boards connects commits, branches, and PRs to work items for traceability from requirement to code. Configure the connection in Azure DevOps (see GitHub and Azure Boards). After linking, use AB#<id> or similar conventions in commits/PRs so state transitions and rollups stay accurate.

3. Design and implement branch strategies and workflows

Design and implement branch strategies and workflows — Microsoft Learn (~1 hr 32 min · 10 units · Advanced).

  • Implement GitHub Flow.

Choosing a workflow

Pick a workflow using team size, release cadence, quality gates, tooling integration, and operational cost. Learn presents a comparison: GitHub Flow fits continuous delivery and simple gates; feature branches isolate work with review before merge; release branches add stabilization for formal releases; forking suits open source and strict access boundaries.

GitHub Flow (typical pattern)

One long-lived main branch stays production-ready. Create short-lived feature branches from main, open pull requests for review and CI, merge when green. This minimizes cognitive load and speeds feedback when tests and deploys are automated.

Merging restrictions and Azure Repos

Enforce branch policies in Azure Repos (required reviewers, build success, linked work items) so merges match your risk profile. The long exercise unit covers version control with Git in Azure Repos end-to-end.

Why pull requests matter

Pull requests are the main place for integrating change safely: they combine review, discussion, automation (build/test), and merge rules. Done well, they spread knowledge, align architecture, and catch defects before mainline.

Azure Repos in the enterprise

Azure Repos integrates with Microsoft Entra ID for identity, supports large repos and high volume, and connects to Azure Boards (work items, traceability), Azure Pipelines (build/validation policies), and Azure Test Plans. Strategic code review

Effective reviews go beyond bug hunting: check architectural consistency, teach patterns, standardize quality, and include security mindset. Give specific, actionable feedback; separate must-fix from suggestions; explain reasoning to grow the team.

Branch policies and automation

Policies can require reviews and builds, link to work items for traceability, and integrate status checks from pipelines or external systems. Use metrics (time to merge, review participation, policy bypasses) to improve the process without drowning velocity.

5. Explore Git hooks

Explore Git hooks — Microsoft Learn (~14 min · 5 units · Advanced).

Prerequisites (Learn): Solid Git workflows; Azure DevOps usage; scripting (Bash, PowerShell, or Python); CI basics; security awareness.

Learning objectives

  • Create hooks that enforce code quality for the team.
  • Run checks before and after commits.
  • Block secrets and unsafe patterns from being committed.
  • Connect hooks with Azure DevOps and pipelines.
  • Share hooks so the whole team uses them.

Why automate on the client

Git hooks run scripts at lifecycle points (e.g. pre-commit, commit-msg, pre-push). They support shift-left: catch formatting, lint, tests, and secrets before code reaches the server—cheaper than fixing in production. In regulated environments, hooks can enforce policy and leave an auditable trail when combined with server-side rules.

Common hook types

  • Pre-commit: Formatting, linters, quick tests, secret scanning, dependency scans.
  • Commit-msg / prepare-commit-msg: Enforce message format and links to work items (e.g. #1234, AB#1234 for Azure Boards).
  • Pre-push: Broader tests, integration checks, or branch naming rules.

Azure DevOps alignment

Hooks complement branch policies and pipelines: local hooks give fast feedback; server-side builds and PR checks are the source of truth for merge. You can use hooks to queue or validate against pipeline definitions (e.g. ensure a validation build for large changes). Standardize hooks via shared install scripts or frameworks so every developer runs the same gates.

6. Plan to foster inner source

Plan to foster inner source — Microsoft Learn (~20 min · 6 units · Advanced).

Prerequisites (Learn): Basic Git and collaboration concepts.

  • Use Git to foster inner source across the organization.
  • Implement fork workflows for collaboration.
  • Choose branches vs forks by scenario.

What is inner source?

Inner source applies open-source practices inside the company: discoverable code, PR-based contribution, and shared ownership—while code stays private. It improves cross-team collaboration, reuse, quality (more reviewers), and speed of innovation.

Branches vs forks

For one team with write access, the usual flow is branch → PR → merge. For another team without write access—or to experiment safely—fork the repo, push to your fork, open a PR back to upstream. Maintainers review and merge. That is the same model as open source, applied internally.

Example: scaling contribution

Large enterprises (including Microsoft’s internal engineering system on Azure Repos) use inner source so any engineer can fork, fix, or extend another team’s component without needing direct write access—reducing silos and duplicate code.

7. Manage and configure repositories

Manage and configure repositories — Microsoft Learn (~48 min · 15 units · Advanced).

Learning objectives

  • Work with large Git repositories.
  • Use VFS for Git where appropriate.
  • Use Git Large File Storage (Git LFS) for big binaries.
  • Clean up repository data.
  • Manage and automate release notes with GitHub.

Why repos get huge

Repositories grow from long history (years of commits) and large binaries (media, builds). Very large clones stress disk, network, and developer patience—techniques like shallow clone (git clone --depth N), single-branch clone, partial clone, and sparse-checkout reduce what each developer needs locally.

VFS for Git and Scalar

VFS for Git virtualizes the working tree so files hydrate on demand; it works with a Git LFS–style client for monorepo-scale trees. Scalar is a Microsoft tool that enables partial clone, background prefetch, sparse-checkout, filesystem monitor, commit-graph, multi-pack-index, and incremental repack—speeding everyday commands on huge repos. Azure Repos can expose the GVFS protocol for compatible clients.

Git LFS

Git LFS stores large files outside the normal Git object store and replaces them with pointers in commits—better for binaries than bloating history. See GitHub’s managing large files.

8. Identify technical debt

Identify technical debt — Microsoft Learn (10 units · Advanced).

Learning objectives

  • Find and manage technical debt.
  • Use code quality tools to improve software.
  • Plan effective code reviews.
  • Understand complexity and quality metrics.

What is technical debt?

Technical debt is the future cost of shortcuts taken today—like financial debt, it accrues “interest”: each hack makes the next change slower and riskier. Causes include deadlines, unclear requirements, weak standards, skipped tests, copy-paste instead of refactor, poor architecture choices, over-engineering, missing docs, and dead code.

Debt reduces velocity, frustrates teams, increases defects and support cost, and can make the org unable to respond to customers. It must be paid down or delivery eventually becomes prohibitively expensive.

Measurement and tools

Automated analysis (quality gates in CI) slows uncontrolled debt. Learn covers tools such as SonarCloud / SonarQube (rules, quality profiles, integration with Azure Pipelines), GitHub Advanced Security (dependency, secret, and code scanning), and other analyzers. Combine metrics (complexity, duplication, coverage) with human review to prioritize fixes.

Code reviews and debt

Reviews should flag architectural drift, missing tests, and risky shortcuts—not only syntax. Consistent standards and refactoring time in sprints prevent debt from compounding silently.
